It covers every mechanic in the MVP scope, the JSBSim deferral rationale, input mapping, port/adapt/drop decisions for every v1 constant, the Cesium coordinate conversion design, per-frame performance budget, UE5-specific risks, and an honest timeline assessment. Mechanic Viability

Each MVP mechanic is assessed against UE5 5.4 Blueprint-first constraints, the 4-week timeline (solo operator), and feel-parity with v1. Verdict codes: VIABLE = implementable within timeline, VIABLE WITH CAVEAT, RISKY.

1. Throttle / Pitch / Roll / Yaw Flight (Arcade)

Verdict: VIABLE — This is the lowest-risk mechanic in v2. UE5's Pawn system is an almost direct analog of v1's game.player state + jet.quaternion manipulation.

UE5 Pawn Pattern: Create a custom Pawn class (BP_AeroFightersPawn) with a StaticMeshComponent for the aircraft body. Override SetupPlayerInputComponent in Blueprint to bind Enhanced Input actions. All flight physics go into Event Tick (BP) using Get World Delta Seconds — this gives the delta-time equivalent to v1's dt parameter.

v1 physics translation to UE5:

  • v1: game.player.speed += (tgtSpd - game.player.speed) * dt * CONVERGE_RATE → UE5: identical lerp on a float variable, applied in Event Tick
  • v1: quaternion-based pitch/roll/yaw via Three.Quaternion → UE5: AddLocalRotation(FRotator) and AddWorldRotation on the Pawn root component, with delta-time scaling
  • v1: forward movement jet.position.addScaledVector(fwd, speed * dt) → UE5: AddActorLocalOffset(ForwardVector * Speed * DeltaTime)
  • v1: gravity subtraction → UE5: AddActorWorldOffset(FVector(0,0,-Gravity * DeltaTime * 100)) (note: UE5 uses centimeters; all v1 meter constants must multiply by 100)

Input Mapping (Enhanced Input): UE5 5.4 uses the Enhanced Input subsystem. Required InputAction assets:

  • IA_Throttle — 1D axis (W/S or gamepad left stick Y)
  • IA_Pitch — 1D axis (arrow Up/Down or gamepad right stick Y)
  • IA_Roll — 1D axis (arrow Left/Right or gamepad right stick X)
  • IA_Yaw — 1D axis (Q/E or gamepad left stick X)
  • IA_FireCannon — bool (Space / gamepad right trigger)
  • IA_BarrelRoll — bool (R or gamepad left bumper)

Feel parity with v1: v1's inverted pitch (pitchDown key = nose up) is a deliberate "simulator-style" inversion. This must be preserved in v2 by negating the pitch input binding value. The PITCH_RATE (1.45 rad/s), ROLL_RATE (2.30 rad/s), and YAW_RATE (0.80 rad/s) port verbatim — divide by pi/180 for degree equivalents if needed. UE5 expects rotations in degrees by default; keep v1 radians internally and convert at the UE5 call site.

BP-first risk: The flight tick runs every frame. With Lumen + Cesium tile streaming also consuming CPU/GPU, BP overhead on the tick is measurable. However, the arcade flight model is arithmetic-simple (no aerodynamic matrices, no lookup tables) — the BP overhead for a handful of float ops and one AddActorLocalOffset call per frame is negligible. This is NOT a C++ migration candidate in MVP.

2. Stall Detection

Verdict: VIABLE — Trivial port. One float comparison per tick.

v1 stall: game.player.stalled = game.player.speed < PLAYER.STALL_SPD (10 m/s threshold). In UE5 this is a single Blueprint boolean variable bIsStalled set in the flight tick. The stall HUD warning is a UMG widget visibility toggle.

Stall thresholds to PORT verbatim: STALL_SPD = 10 m/s = 1000 cm/s, MIN_SPD = 8 m/s. These felt correct in v1 and the Inhauma photoreal terrain does not change the aerodynamic model.

Stall behavior in v1: When stalled, lift is zero and gravity overcomes the forward vector's vertical component. This is simple to replicate: in the flight tick, only add the lift compensation if !bIsStalled.

3. Crash Detection

Verdict: VIABLE WITH CAVEAT — Cesium tile mesh collision is not trivial. Recommend a hybrid strategy.

v1 crash detection: checkTerrainCollision samples island height at the player XZ position and crashes if jet.position.y < heightAt + MOUNTAIN_BUFFER. This works because v1's terrain is procedurally generated with a known mathematical height function.

v2 challenge: Cesium for Unreal streams photorealistic 3D tiles. These tiles are UCesiumGlobeAnchorComponent actors with complex mesh geometry. UE5 physics collision against streaming Cesium tile meshes is supported but requires tile collision cooking to be enabled — this has a streaming performance cost and can cause LOD popping artifacts where tile collision does not match visual LOD.

Recommended hybrid strategy:

  1. Analytic ground plane (MVP primary): Approximate terrain height at the aircraft's WGS84 lat/lon using a lightweight height query. Cesium for Unreal exposes UCesium3DTileset::FindGeospatialGroundAltitude which samples the DEM height from the tileset without requiring full collision cooking. Use this as the crash threshold check: if aircraft altitude (from CesiumGeoreference) < terrain height + buffer, trigger crash.
  2. Cesium tile collision (deferred to v2.1): Full mesh collision for terrain-hugging flight is a post-MVP feature. Mark as DEFER in TASKS.
  3. Sea crash (MVP simple case): Aircraft altitude < 0m MSL = instant crash. No collision query needed.
Risk: FindGeospatialGroundAltitude is an async query in some versions of Cesium for Unreal. If asynchronous, crash detection must use the last-known altitude value and tolerate one frame of lag. This is acceptable for arcade gameplay but must be tested early (Week 1).

4. Cannon (12.5 r/s Vulcan)

Verdict: VIABLE — Straightforward Actor pooling in UE5. BP-first is adequate.

v1 cannon: CANNON.RATE = 0.08s between shots (12.5 shots/s), BULLET_SPD = 110 m/s, BULLET_LIFE = 2.0s, bullet is a cylinder mesh pooled in bulletPoolPlayer. Collision is a simple distance check against target hr2 (hit radius squared).

UE5 port design:

  • Projectile Actor Pool: Create BP_CannonRound as a AActor with a ProjectileMovementComponent. Pre-spawn a pool of 30 actors (12.5/s * 2.0s life = 25 maximum in-flight, pool of 30 gives headroom). On fire: get from pool, set position to muzzle location, enable ProjectileMovementComponent with initial velocity = forward vector * BULLET_SPD * 100 (cm/s).
  • Hit detection: ProjectileMovementComponent supports bShouldBounce=false and OnProjectileStop delegate. Alternatively, use a SphereTraceByChannel per projectile per tick (simpler, more controllable for arcade). Recommended: use ProjectileMovementComponent's built-in overlap events with the AA gun's BoxCollision component. This is the idiomatic UE5 approach and avoids manual distance checks.
  • Fire rate enforcement: A float CannonCooldown variable in the Pawn, decremented by DeltaTime, prevents firing while > 0. Reset to 1.0f / FIRE_RATE = 0.08s on each shot.
  • Muzzle position: Attach a SceneComponent ("Muzzle") to the aircraft mesh at the cannon position. Spawn projectiles from this component's world location.

FX hookup: On projectile hit, spawn a Niagara particle system (NS_CannonImpact) at the hit location. For MVP, a simple sprite burst system. Do not block on FX — spawn asynchronously via UNiagaraFunctionLibrary::SpawnSystemAtLocation.

AC-V2-04 verification path: The test "125 ± 2 shots in 10 seconds" can be validated by incrementing a counter in the OnFire Blueprint event and exposing it via a debug command.

5. AA Gun (Static Enemy Archetype)

Verdict: VIABLE — Recommend dumb-fire for MVP. Full tracking is a post-MVP feature.

v1 AA gun behavior:

  • Static mesh, octagonal base + turret + twin barrels
  • In range (<220m): rotates to face player (instant yaw snap), fires at BASE_INTERVAL = 1.7s
  • Fires spawnBullet(dir = normalize(playerPos - gunPos)) — a perfect aim-at-player shot
  • Enemy bullet speed: 56 m/s (hardcoded in projectiles.js)

MVP recommendation — dumb-fire with aim bias: For MVP, implement the AA gun as a Blueprint Actor (BP_AAGun) that:

  1. On Event Tick, checks player distance via GetDistanceTo.
  2. If within RANGE = 22000 cm (220m * 100): snap-rotate toward player using FindLookAtRotation + SetActorRotation. This reproduces v1's instant yaw.
  3. Fire via a TimerByEvent at 1.7s interval (jittered by +/- 0.4s for variety).
  4. Enemy projectiles: same BP_CannonRound actor pool, different material (red tracer per v1 COLORS.bulletEnemy).

Why dumb-fire (not full tracking): v1's AA fires a straight bullet aimed at the player's current position, not a homing projectile. This is already "dumb-fire" in v1. A "full tracking" version would require predictive aiming — that is a post-MVP enhancement. MVP fidelity = v1 behavior = instant aim + linear projectile.

Hit state and destruction: BP_AAGun exposes a CurrentHP variable (v1: hp=6). On cannon hit (via overlap event), decrement HP. At 0: play destruction Niagara effect, hide mesh, award score. Score: 250 (v1 value, port verbatim). The destruction is permanent for the session (no respawn in MVP).

Placement: For MVP, place exactly one AA gun at a fixed WGS84 coordinate in the Inhauma playable area using the CesiumGeoreference system (see Cesium section below). Recommend placing on a recognizable landmark — a hilltop near Inhauma. This satisfies AC-V2-05. In-editor placement is the simplest approach for MVP (see OQ-12 answer below).

JSBSim Feasibility

Per locked decision D4 (grill Q5), JSBSim FDM integration is explicitly deferred beyond MVP. This section documents the future integration path so the synthesis SPEC can make the deferral explicit and the PLAN can eventually pick it up.

Decision: JSBSim is NOT in MVP scope (D4, LOCKED)

Rationale (game-developer authority): The arcade flight model in v1 produces responsive, fun controls. The MVP's goal is "minimum-viable-combat" — photoreal terrain + cannon + AA gun. Adding JSBSim in MVP would require: (1) sourcing an F-35 FDM coefficient file, (2) integrating JSBSim as a DLL or plugin, (3) bridging WGS84 coordinates between Cesium and JSBSim's NED frame, (4) tuning and validating aerodynamic behavior against a photoreal environment. This alone is a multi-week effort that competes directly with the 4-week total timeline. Deferral is correct.

JSBSim Required Version and License

Version: JSBSim 1.2.x (latest stable as of 2025). License: LGPL-2.1 with a classpath exception that permits linking without contaminating the game binary under proprietary terms. Compat with UE5: confirmed for UE5.3/5.4 via community reports — the library has no UE5-native plugin and must be integrated manually.

Integration Path into UE5

Three options exist, in ascending complexity:

OptionMethodEffortRiskRecommendation
A Compile JSBSim from source as a static library, link in UE5 plugin (C++ only, no BP exposure) 2-3 days initial setup + CMake/UBT integration Medium — UBT (Unreal Build Tool) does not speak CMake natively; use a wrapper module ThirdParty/JSBSim/JSBSim.Build.cs Preferred for v2.1
B Bind JSBSim as a shared DLL (.so / .dll) loaded at runtime via FPlatformProcess::GetDllHandle 3-5 days; requires Windows + Linux DLL builds separately High — DLL ABI differences between Linux/Windows; harder to debug Avoid
C Out-of-process via socket: JSBSim runs as a subprocess, game sends inputs and receives state via UDP 1-2 days for basic socket bridge; JSBSim supports this via its `Socket` FDM interface High — latency, process lifecycle management; not suitable for production game Avoid (OK for prototyping only)

Migration trigger for JSBSim landing in v2: JSBSim should be introduced when ALL of the following are true:

  1. The arcade model has been played by the operator's target audience (Tauan) and feedback confirms the desire for more realistic flight behavior
  2. The UE5 project has shipped at least one stable release (v2 MVP) with a working build pipeline for both Windows and Linux
  3. The performance budget (16.6 ms/frame) has headroom of >4ms after Cesium tile streaming + cannon + AA gun FX
  4. The operator explicitly scopes JSBSim as a SPEC-approved feature in a v2.1 or later release

F-35 FDM Coefficient Sourcing

The F-35 is not open-source in the JSBSim aircraft database. Options:

  • Hand-tuned coefficients: Start from the JSBSim generic jet aircraft template (737.xml or A320.xml), tune lift/drag curves to match the F-35's published performance envelope (Mach 1.6 max speed, ~9g turn capability at combat weight, ~270 knots minimum controllable). This is the practical path for a game.
  • OpenVSP + DATCOM: Approximate F-35 geometry in OpenVSP, export to DATCOM to generate aerodynamic coefficients. Produces plausible but not certified data. Acceptable for a game.
  • Commercial license: Not applicable to a personal game project.

Cesium-to-Pawn Coordinate Conversion

This is the most technically novel aspect of v2. v1 operates in a flat Three.js local-coordinate space. v2 operates in photoreal WGS84 geography. All movement, collision, and placement logic must work in the correct coordinate frame.

Coordinate Systems in Play

SystemUnitUsed for
WGS84 (lat/lon/alt) degrees / meters MSL Cesium tile anchor, AA gun geo-placement, player spawn position
ECEF (Earth-Centered Earth-Fixed) meters (cartesian, geocentric) Cesium internal representation; not directly used in gameplay code
UE5 World (local UE coordinates) centimeters Physics, rendering, actor transforms, projectile movement
ENU (East-North-Up, local tangent plane) meters (local) Flight physics vectors (forward, up); this is the natural frame for arcade flight

CesiumGeoreference — The Bridge

Cesium for Unreal provides a ACesiumGeoreference actor that defines the mapping between WGS84 and UE5 world coordinates. The georeference origin is set to the Inhauma MG reference point (lat=-19.47, lon=-44.46, alt=800m). At the origin, UE5 coordinates (0,0,0) map to the WGS84 origin.

Recommended conversion layer: use CesiumGeoreference's built-in helpers. Do NOT write a custom WGS84-to-ECEF-to-UE5 conversion. Cesium for Unreal exposes:

  • ACesiumGeoreference::TransformLongitudeLatitudeHeightToUnreal(lat, lon, alt) → returns FVector in UE5 world centimeters
  • ACesiumGeoreference::TransformUnrealToLongitudeLatitudeHeight(FVector) → returns FVector(lon, lat, alt)
  • UCesiumGlobeAnchorComponent — attach to any Actor to auto-convert between geo and UE5 coordinates every frame

API Design Sketch

// BP callable — place AA gun at geo coordinate
// Called from BP_LevelManager on BeginPlay

// Given: lat=-19.47, lon=-44.46, alt=820m (20m above Inhauma reference)
FVector uePos = GeoRef->TransformLongitudeLatitudeHeightToUnreal(
    FVector(-44.46, -19.47, 820.0)  // Cesium uses (lon, lat, alt) order
);
AAGun->SetActorLocation(uePos);

// Player spawn: spawn pawn at WGS84 coordinate
FVector spawnPos = GeoRef->TransformLongitudeLatitudeHeightToUnreal(
    FVector(-44.46, -19.47, 1100.0)  // 300m above reference altitude
);
GetWorld()->SpawnActor<AAeroFightersPawn>(SpawnClass, spawnPos, FRotator::ZeroRotator);

// Crash detection: convert current pawn position to altitude
FVector geoPos = GeoRef->TransformUnrealToLongitudeLatitudeHeight(
    Pawn->GetActorLocation()
);
float altitudeMSL = geoPos.Z;  // meters above mean sea level
if (altitudeMSL < TerrainHeightMSL + MOUNTAIN_BUFFER_M) { TriggerCrash(); }

Flight Physics Frame

The arcade flight model uses local axes (forward, up, right). In UE5, the Pawn's local X axis is forward. The conversion between local movement and world movement is handled by AddActorLocalOffset (no explicit frame conversion needed for movement). The only frame that requires attention is gravity: gravity must be applied in the world Z-down direction, not local Z-down — use AddActorWorldOffset(FVector(0, 0, -GravityAccel * DeltaTime)).

OQ-07 Answer (Time of Day for Screenshot-Diff Determinism)

Proposed answer: Lock sky time of day to a fixed value at test startup. In UE5, set the Directional Light rotation and Sky Atmosphere sun pitch via Blueprint before the first frame renders. Screenshot-diff tests should pass a ?timeOfDay=noon launch parameter (analogous to v1's ?testMode=1 pattern) that sets sun azimuth to 180° and elevation to 60°. This eliminates cloud/shadow variability from diff tests.

OQ-11 Answer (Aircraft Model Source)

game-developer position: For MVP, build a simplified UE5 procedural mesh using UE5's ProceduralMeshComponent (C++ or Blueprint). The v1 F-35 geometry (nose cone, fuselage, trapezoidal wings, V-tails) can be approximated using Box/Cone meshes assembled as a Blueprint Actor with attached Static Mesh Components — the same approach as v1 but in UE5 Static Mesh primitives. This preserves the no-external-assets principle from the foundation spec for the MVP. A higher-quality mesh (Quixel or custom FBX) can be introduced in v2.1 if the game-designer proposes and the operator approves.

OQ-12 Answer (AA Gun Placement)

game-developer recommendation: Use in-editor placement for MVP. Place BP_AAGun in the UE5 level editor, set its UCesiumGlobeAnchorComponent longitude/latitude/height properties in the Details panel. This gives the operator visual confirmation of placement on the Cesium tile map. Store the coordinate in a Data Asset (DA_AeroFightersV2Config) so tests can read and assert the expected geo-position programmatically. No need for a JSON file in MVP; Data Assets are UE5-native and BP-accessible.

Per-Frame Performance Budget

Target: 60 FPS @ RTX 3060 1080p = 16.6 ms per frame (D6, locked). The RTX 3060 has 12 GB GDDR6, 192-bit bus, 360 GB/s bandwidth — a capable mid-range card for UE5.

SystemEstimated ms (RTX 3060 1080p)NotesMitigation if over budget
Cesium Tile Streaming + Rendering 7.0 – 9.0 ms Dominant cost. Includes tile decode, mesh upload, Nanite render (if tiles support Nanite) or standard mesh pass. Google Photorealistic 3D Tiles are glTF — Cesium for Unreal converts them to UE5 Static Meshes at runtime. Lumen GI on top of dynamic tile geometry adds significant cost. Disable Lumen for tile geometry (use indirect lighting cache). Limit tile streaming radius to 10 km instead of 20 km. Reduce max simultaneous tile count via Cesium.MaximumSimultaneousTileLoads.
UE5 Lumen GI + Reflections 2.0 – 4.0 ms Lumen ray-tracing budget. At 1080p with Lumen quality "Medium" on RTX 3060, expect 2-4ms. Hardware Ray Tracing mode would be higher. Use Lumen "Low" quality preset for MVP. Disable screen-space reflections on Cesium tile surfaces (tiles are opaque terrain — reflections not needed).
Flight Physics Tick (BP) 0.1 – 0.3 ms Arcade flight is 10 float operations per tick. Blueprint overhead is real but the math is trivial. Even at 60 FPS, the BP tick for one pawn is <0.3ms. Promote to C++ only if profiler shows >2ms (clearly not a flight physics issue at that point).
Cannon Projectiles (pool of 30) 0.2 – 0.5 ms ProjectileMovementComponent for 25 active projectiles. UE5 batch-processes physics at fixed intervals. At 30 projectiles the physics thread cost is negligible. Disable physics simulation on projectiles — use simple AddActorWorldOffset per tick instead of ProjectileMovementComponent if overhead appears.
AA Gun AI Tick (1 actor) 0.05 ms Distance check + rotation + timer. Negligible for 1 actor. N/A for MVP.
Niagara FX (cannon impacts, jet exhaust) 0.5 – 1.5 ms Jet exhaust: 1 persistent Niagara system. Cannon impacts: short-lived burst systems, max 3 simultaneous. Objections (Cross-Domain)

OBJ-1 [to software-architect] — BP-first is correct for flight tick (D12 assertion)

Domain: Mechanics / UE5-specific — game-developer has tiebreaker authority.

Anticipated objection: The software-architect may argue that the flight physics tick is a C++ hot path candidate due to the 60 FPS performance requirement (D6).

game-developer position with evidence: The arcade flight model performs ~10 float operations per tick: two lerps, three rotations, one translation, and three float comparisons. On a modern CPU at 60 FPS, this is approximately 0.01-0.05ms of compute regardless of whether it runs in BP or C++. The RTX 3060 performance budget is consumed by Cesium tile streaming and Lumen — not by arcade flight arithmetic. Blueprint nativization at packaging time eliminates the BP interpretation overhead for Shipping builds. Therefore, promoting flight physics to C++ in MVP is premature optimization and increases development complexity without measurable benefit. The migration trigger (D15 / OQ-02) should be set at BP tick overhead >2ms per frame as measured by the UE5 Profiler's Blueprint tab — this threshold will not be crossed by simple arcade flight math.

OBJ-2 [to game-designer] — Aircraft mesh should be procedural primitives in MVP (OQ-11)

Domain: Asset sourcing — game-designer has primary authority on visual/asset choices; game-developer objection requires timeline evidence.

game-developer position with evidence: The game-designer may propose a Quixel Megascans aircraft mesh or a downloaded FBX for the F-35. This report's position (already stated in OQ-11 section): use UE5 primitive Static Mesh components assembled in Blueprint for MVP. Evidence: (1) The 4-week timeline with a first-time UE5 operator leaves no time for asset import pipeline troubleshooting (LOD setup, collision hull, material slot assignments, UV mapping). (2) v1 used a procedural Three.js mesh and this is consistent with the foundation spec's "no external assets" philosophy, which the operator should explicitly override in the SPEC if importing an FBX. (3) A primitive-mesh aircraft is sufficient to validate AC-V2-03 (flight), AC-V2-04 (cannon), and AC-V2-05 (AA gun hit). Visual quality can be upgraded in v2.1 with proper asset pipeline time.

OBJ-3 [to game-tester] — Perf gate must allow headless AND windowed modes (C6 from discovery)

Domain: Test strategy — game-tester has primary authority; game-developer objection with profiler evidence.

game-developer position: A headless (no GPU) UE5 perf gate does not exercise Cesium tile streaming or Lumen — the two systems that dominate the 16.6ms budget.
