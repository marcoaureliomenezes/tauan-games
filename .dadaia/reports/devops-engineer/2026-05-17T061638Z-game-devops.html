DevOps Review — aero-fighters-v2: Photorealistic Inhauma v1

This report covers all CI/CD, build, deployment, and secret-management decisions required for the aero-fighters-v2 UE5.4 release alongside the existing v1 (Three.js) infrastructure. Every section maps directly to a locked operator decision from the discovery report. Cross-domain objections are isolated in the final section.

Pipeline philosophy applied: this is a personal game project for a solo operator. Every step must earn its place. A 4-week timeline means CI must be minimal-but-correct, not aspirationally complete. Complexity is deferred; correctness is mandatory from day 1.

Dev workflow: .env.local protocol

A committed aero-fighters-v2/.env.example documents the required variable. The actual .env.local is gitignored and must never be committed:

# aero-fighters-v2/.env.example
# Copy this to .env.local and populate from 1Password:
#   op item get "aero-fighters-v2/google-maps-tiles-api-key" --field credential
GOOGLE_MAPS_TILES_API_KEY=<get-from-1password>

# Set to 1 to use the offline tile cache (no live API calls, no cost)
# 0 = live fetch from Google Maps Tiles API
# Default for dev: leave 1 unless you need to refresh the cache
CESIUM_OFFLINE_TILES=1
Offline cache as default: Per discovery conflict prediction C7 (devops wins on cost enforcement), the offline tile cache must be the default dev workflow once the cache is populated. Live tile fetch is opt-in via CESIUM_OFFLINE_TILES=0. See Offline Tile Cache section below.

How CI receives the key

The key is stored as a repository-level GitHub Secret (not org-level). In v2 workflows:

env:
  GOOGLE_MAPS_TILES_API_KEY: ${{ secrets.GOOGLE_MAPS_TILES_API_KEY }}

This secret is never referenced in ci-v1.yml or pages.yml, ensuring v1 CI cannot accidentally trigger tile API calls or expose the secret scope.

How the release binary embeds the key

For MVP, embed at build time is the correct strategy over a runtime config fetch. Runtime fetch would require a config server (infrastructure not justified for a personal project).

  1. CI injects GOOGLE_MAPS_TILES_API_KEY as an environment variable during the UE5 Shipping build step (via the -set: argument to RunUAT or by pre-writing to a project config file before invoking the build command).
  2. The UE5 project (game-developer's responsibility) reads this variable during the cook phase and bakes the value into the packaged DefaultGame.ini or Cesium config.
  3. The resulting zip contains the key and functions without any runtime config dependency.
Risk: embedded key in a public GitHub Release zip is extractable. For a personal project with a $20/month hard cap and billing alerts, this is an accepted risk. The worst case before the alert fires and the key is rotated is a $20 overcharge. If the game's user base grows, move to a runtime-fetch architecture with key rotation. This is explicitly post-MVP scope.

Offline tile cache strategy (OQ-05)

DevOps-side proposal (implementation is game-developer's responsibility per the objection in the final section):

  • Layout: aero-fighters-v2/.tile-cache/ — gitignored. A .tile-cache/.gitkeep keeps the directory tracked without content.
  • Size cap: 5 GB (covers the 20 km radius around Inhauma at useful LOD — exact figure is a PLAN deliverable requiring measurement). The populate script checks available disk space before writing and aborts if remaining space would drop below 10 GB.
  • Eviction policy: LRU by file modification time. The populate script prunes tiles older than 30 days when total size exceeds the cap.
  • Populate command: make tile-cache-populate — a Makefile target wrapping the Cesium offline cache fetch script (implemented by game-developer using the Cesium for Unreal offline cache API). Requires .env.local to be populated with the API key.
  • CI behavior: CI does not populate the offline cache. CI screenshot-diff tests use the live API (counted against the $20 cap). This is acceptable because screenshot-diff tests run only on tagged release builds, not every PR — estimated CI tile spend: $0.50–$2.00 per release tag (see Cost Monitoring section).

Runner options analysis (resolves OQ-08)

INSUFFICIENT GitHub-hosted ubuntu-latest / windows-latest

Specs: 16 GB RAM, 4 cores, ~14 GB disk free after OS (Windows) or ~30 GB (Linux).

Verdict: Structurally insufficient for UE5.4 + Cesium Shipping builds. The disk constraint alone (14 GB on Windows) eliminates this option — the UE5 engine alone exceeds it. RAM (16 GB) is also below the minimum for plugin-heavy Shipping builds.

TOO EXPENSIVE GitHub larger runners (16-core, 64 GB RAM)

Specs: 64 GB RAM, 500 GB disk, 16 cores. Billed at $0.256/min (Linux) or $0.512/min (Windows).

Cost per build: A UE5.4 Shipping build (project code + Cesium, engine pre-cached — not possible on GitHub-hosted) takes 60–120 min. At $0.512/min for Windows: $30–$60 per platform, $60–$120 per release (both platforms). This exceeds the $20 tile API cap and is unsustainable for a solo personal project with no revenue.

Additional problem: GitHub-hosted runners cannot cache the UE5 engine binaries between runs (the cache limit is 10 GB; UE5 engine binaries exceed 50 GB). Every run would require a fresh engine download + compilation — 4–6 hours per build on the largest available runners.

RECOMMENDED Self-hosted runners on operator hardware

Cost: $0 marginal (electricity only, ~$5–15/month for active build sessions).

Advantages: Full control over engine pre-installation, DDC persistence, GPU access for test jobs, no disk constraints, no per-minute billing.

Setup cost: ~2–4 hours one-time to register runners and install UE5.4. This must happen in Week 1 of the 4-week sprint — it is a hard dependency for all v2 CI.

Constraints: Operator's machine must be online when CI triggers (tag pushes trigger release builds). For a solo operator, this is acceptable — they control when they push tags.

Runner label scheme

  • self-hosted, windows, ue5-builder — Windows Shipping builds + Windows GPU tests
  • self-hosted, linux, ue5-builder — Linux Shipping builds + Linux GPU tests

If the operator has only a Windows machine, Linux can run in WSL2 registered as a second runner, or a Linux VM with GPU passthrough. If Linux CI proves impossible to set up within Week 1 constraints, this is the D17 if-slip trigger point — not week 3 (Cesium rendering), but week 1 (runner availability).

Caching strategy

  • Engine version pin: A committed file aero-fighters-v2/.ue5-version contains the exact UE5 minor version (e.g., 5.4.4). CI reads this and verifies the pre-installed engine on the runner matches before building. Mismatch causes a clear failure with an actionable error message rather than a cryptic build failure.
  • Cesium plugin pin (OQ-10): The Cesium for Unreal plugin is pinned by commit hash in a committed file aero-fighters-v2/.cesium-version (or as a git submodule). The software-architect proposes the compatible version; devops owns the pin format. See objection in final section.
  • DDC: Persisted as a named directory on the self-hosted runner's local disk. Not uploaded to GitHub Actions cache (too large — 30–50 GB vs. 10 GB cache limit). Survives across runs automatically. Manual invalidation when engine version pin changes.
  • GitHub Actions artifact cache: Used only for small inter-job data (e.g., build metadata passed from build jobs to the publish job). Not used for engine or DDC.

Distribution Pipeline (GitHub Releases)

LOCKED D11: GitHub Releases zip, attached to git tag. CI builds .exe + Linux binary. No itch.io, no Steam.

Tag naming convention (resolves OQ-13)

This is a devops-engineer decision per the Decision Authority Matrix (CI/build tie-breaker):

  • v1 tags: aero-v1-<semver> — e.g., aero-v1-1.3.0
  • v2 tags: aero-v2-<semver> — e.g., aero-v2-0.1.0

The tag prefix is the unambiguous trigger discriminator between v1 and v2 workflows. No other naming scheme (date-based, commit-hash-based) is considered — semver with the game prefix is the most human-readable and toolchain-compatible format.

Full distribution flow

  1. Operator confirms the release is ready, then runs: git tag aero-v2-0.1.0 && git push origin aero-v2-0.1.0
  2. release-v2.yml triggers on push: tags: aero-v2-*.
  3. Workflow checks out the tag with fetch-depth: 0 for changelog generation.
  4. Matrix job fans out two parallel builds: build-windows on [self-hosted, windows, ue5-builder] and build-linux on [self-hosted, linux, ue5-builder].
  5. Each build job: verifies .ue5-version matches the pre-installed engine. Verifies .cesium-version pin. Injects GOOGLE_MAPS_TILES_API_KEY from GitHub Secret into the build environment.
  6. Each build job: invokes RunUAT.sh BuildCookRun (Linux) or RunUAT.bat BuildCookRun (Windows) with -shipping, -platform=Win64 / -platform=Linux, and the project .uproject path.
  7. Each build job: creates a zip artifact: aero-fighters-v2-win64-aero-v2-0.1.0.zip or aero-fighters-v2-linux64-aero-v2-0.1.0.zip.
  8. Each build job: generates SHA256 checksum: sha256sum aero-fighters-v2-win64-aero-v2-0.1.0.zip > aero-fighters-v2-win64-aero-v2-0.1.0.zip.sha256
  9. Each build job: uploads both files as GitHub Actions artifacts (zip + .sha256).
  10. publish-release job runs on ubuntu-latest after both build jobs complete (needs: [build-windows, build-linux]).
  11. Publish job downloads all 4 artifacts. Generates changelog from git log <prev-aero-v2-tag>..HEAD --pretty="- %s (%h)".
  12. Publish job creates the GitHub Release: gh release create aero-v2-0.1.0 --title "aero-fighters-v2 v0.1.0" --notes "$CHANGELOG" aero-fighters-v2-win64-*.zip aero-fighters-v2-win64-*.sha256 aero-fighters-v2-linux64-*.zip aero-fighters-v2-linux64-*.sha256
  13. Release notes include: version, platform matrix, SHA256 checksums table, known issues section, and a note that the API key is embedded (for transparency with the operator; end-user-facing notes do not need to mention the key).

Per CLAUDE.md, Hermes is the operator's Telegram bot running on the VPS. Integration path:

  2. A Cloud Run job (or lightweight Cloud Function) subscribed to that topic forwards the alert as a Telegram message to the operator via the Hermes gateway or directly via the Telegram Bot API.

CI tile API budget control

CI screenshot-diff tests that make live tile requests run only on tagged release builds (release-v2.yml), not on every PR (ci-v2.yml). For a solo operator releasing approximately once per month, this limits CI tile spend to one render session per release. Estimated cost per session: $0.50–$2.00 for 4+ geo-poses in the 20 km radius. This leaves $18–$19.50 per month for operator dev work — comfortable within the $20 cap.

What to do when the cap is hit

  1. GCP Console > APIs & Services > Credentials > Select key > Disable.
  2. Continue dev work using the offline tile cache (CESIUM_OFFLINE_TILES=1 in .env.local).
  3. At the start of the next billing cycle, re-enable the key in GCP Console.
  4. If the overage was due to a CI run, audit which workflow job triggered unexpected tile fetches and add a guard to prevent recurrence.

Path filter isolation guarantee

Path filters ensure that CI runs only the jobs relevant to what changed:

  • Changes to aero-fighters-v2/** only: v2 compile-check runs, v1 Playwright does NOT run.
  • Changes to aero-fighters/** or tests/** only: v1 Playwright runs, v2 compile-check does NOT run.
  • Changes to both (e.g., cross-cutting refactor, package.json update): both run in parallel — still isolated, just concurrent.

This guarantees that v2 development cannot accidentally slow down or break v1 CI, and vice versa.

Secrets isolation

GOOGLE_MAPS_TILES_API_KEY is referenced only in ci-v2.yml and release-v2.yml. It is absent from ci-v1.yml and pages.yml. This prevents any v1 CI run from incurring API costs or accidentally exposing the key.

v1 caching (preserve as-is)

The existing Playwright browser cache pattern (keyed on hashFiles('package-lock.json'), stored at ~/.cache/ms-playwright) is correct. Copy verbatim into ci-v1.yml.

v2 caching (new, self-hosted strategy)

Engine binaries and DDC persist on the self-hosted runner's local disk across all runs. GitHub Actions cache is not used for v2 build artifacts (size exceeds limits). Only small inter-job outputs (metadata, checksums) use the Actions artifact upload/download mechanism.

OBJECTION into game-tester Screenshot-diff CI on every PR will exhaust the $20/month tile budget

Evidence: Google Maps Tiles API pricing: ~$0.004 per session tile request. A single screenshot-diff render at useful LOD for a 20 km radius scene requires 100–500 tile requests (based on Cesium's typical request volume for a medium-altitude view). Four geo-poses per test run = 400–2000 requests = $1.60–$8.00 per CI run. If screenshot-diff runs on every PR during a 4-week active sprint (2–5 PRs/day), monthly CI tile spend reaches $96–$1,200 — far exceeding the $20 cap.

Proposed resolution (devops wins on CI infrastructure): Screenshot-diff tests (AC-V2-06) run only on tagged release builds. PRs get compile-check only. The synthesis SPEC should encode this as a constraint on when AC-V2-06 is validated, so game-tester designs the harness to be triggered exclusively via the release workflow.
