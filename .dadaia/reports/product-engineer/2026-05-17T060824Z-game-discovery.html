Findings

F1. Engine ladder transition is legitimate and authorized

The repository's specs/foundation/SPEC.md explicitly anticipates a Degrau 3 (UE5) jogo named aero-fighters-v2/ in the ladder table (line 30). This release is therefore not a foundation amendment — it executes a slot already reserved by the constitution. No foundation rewrite needed.

Foundation slotThis release fills it with
Degrau 3 engineUnreal Engine 5.4 (Blueprints + C++)
Degrau 3 game directoryrepos/tauan-games/aero-fighters-v2/ (NOT to be created in discovery)
Distribution policy gap Foundation says "build pipeline ainda a definir" — this release closes it (GitHub Releases zip, CI-built .exe + Linux binary, attached to git tag).

F2. Scope is "minimum-viable-combat", not "flight-only"

Grill answers Q1 and Q9 jointly redefine the MVP. Q1 said "flight + cannon + 1 AA gun" which contradicts a literal "flight-only" reading. The reconciled scope:

MVP scope (locked): Photoreal flight over 20 km radius around Inhaúma MG + cannon (12.5 rounds/sec, ported from v1) + exactly one enemy archetype (static AA gun, ported from v1 design). Full arcade-combat surface (multiple enemy types, missiles, bombs, mission system) deferred to v2.1+.

F3. v1 (Three.js) and v2 (UE5) evolve in parallel — this is a CI concern

Grill answer Q10 locks parallel evolution: v1 keeps receiving features while v2 is built. The existing Playwright suite (18 ACs for v1 per repos/tauan-games/AGENTS.md) must remain green throughout the entire v2 development cycle. This creates a CI matrix obligation:

  • v1 lane: Node + Playwright on aero-fighters/ — unchanged
  • v2 lane: UE5.4 headless + Cesium plugin + screenshot-diff tooling — new
  • Shared CI runner constraints: GPU availability for UE5 builds vs. CPU-only for Playwright

F4. Geographic + tile streaming budget = single largest external dependency

Grill Q3 + Q4 fix the playable region (20 km radius around (-19.47, -44.46)) and the Google Maps Tiles API spend cap (USD 20 / month, hard). The combination defines the photoreal acceptance bar's economic ceiling and forces an offline tile-cache strategy for dev workflow.

F4 implication: tile cost is per-byte-served, not per-request. The dev cache, CI runs (screenshot-diff in F8), and operator playtesting all draw from the same budget. Cost-control telemetry must be in place before week 1 of implementation, not after.

F5. Flight model is intentionally arcade for MVP

Grill Q5: hybrid policy. MVP uses v1-style arcade physics (throttle / pitch / roll with fixed coefficients, no aerodynamic surfaces, no fuel model, no atmospheric density). JSBSim FDM integration is explicitly deferred to a later release — out of MVP scope. This is a major risk-reducer: it means no Cesium-Georeferencing-to-FDM coordinate transformation work in MVP.

F6. Dual primary platform commitment doubles CI/test work

Grill Q6: both Windows x64 native and Linux x64 native are primary targets, not "Linux best-effort via Proton". This decision substantially increases scope:

  • Two binary artifacts per tag (.exe and Linux ELF)
  • Cesium for Unreal supports Linux but with smaller test coverage than Windows
  • Screenshot-diff baselines must be generated and validated on both platforms (text rendering, shader compiler differences, driver path differences)
  • CI must have both a Windows runner with a GPU and a Linux runner with a GPU

F7. Performance floor is a public commitment

Grill Q7: RTX 3060 @ 60 FPS @ 1080p is the public floor. This becomes a Non-Functional Requirement enforced by automated perf tests, not a soft target. The 3060 has 12 GB VRAM and Ampere architecture — UE5.4 + Lumen + Cesium tiles + Niagara FX should fit, but the budget needs explicit modeling by the architect.

F8. Acceptance bar for "photorealism" is objective (screenshot-diff)

Grill Q8: photoreal quality is gated by automated screenshot-diff vs Google Earth reference imagery. This is unusual and strong — it means:

  • Reference imagery must be captured and frozen (license/derivation status needs review)
  • Diff tolerance (per-pixel %, perceptual hash, SSIM threshold) is undefined — deferred to PLAN
  • The diff harness lives in tests/ and must run on both Windows and Linux CI
  • The harness is itself a feature with non-trivial scope (rendering at fixed geo-poses, shader determinism, sky/lighting time-of-day fix)

F9. Solo team, BP-first policy, 4-week timeline — these compound

Grill Q11 (4 weeks), Q14 (BP-first), Q15 (solo) combine into the single largest delivery risk on this release. See "Tensions" subsection below for explicit recommendations.

F10. Tensions explicitly surfaced

Tension T1 — Scope vs identity (Q1 vs Q2)

Q1 framed MVP as "flight-only refined to cannon + 1 AA gun". Q2 stated long-term identity is "arcade combat (v1 style)". The risk is that "MVP is flight-only" gets quoted in isolation downstream. Resolution: SPEC and all specialist reports must consistently say "MVP is the minimum-viable-combat slice of the arcade-combat long-term identity". No specialist may treat MVP as flight-only.

Tension T2 — Timeline aggression (Q11)

Four weeks from zero to a playable demo that includes: first-ever UE5 project for the operator, first-ever Cesium plugin integration, Linux + Windows native binary CI, and an automated photoreal screenshot-diff harness. This is aggressive to the point of being a risk-on commitment.

Recommendation to synthesis stage: SPEC must include an explicit "if-slip" decision rule. Suggested wording: "If by end of week 3 Cesium tiles do not render at 60 FPS on Linux, drop Linux from MVP and ship Windows-only; revisit Linux in v2.0.1." The operator should validate this fallback or supply a stricter rule before SPEC approval.

Tension T3 — Parallel v1 evolution introduces CI matrix work (Q10)

Continuing to evolve v1 while building v2 means the existing Playwright suite must stay green throughout. devops-engineer must explicitly plan: shared workflow vs. split workflows, shared runner pool vs. separate, secrets isolation (the GCP key is v2-only), and tag-naming convention that distinguishes v1 from v2 builds (e.g., aero-v1-* vs aero-v2-*).
