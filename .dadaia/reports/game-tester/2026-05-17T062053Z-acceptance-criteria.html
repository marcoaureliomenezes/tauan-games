UE5 Test Strategy

Four candidate frameworks were evaluated against the specific constraints of this project: Cesium tile non-determinism, Blueprint-first hot paths, solo operator, no server-side GPU on standard CI runners, 4-week timeline.

Candidate A — UE5 Functional Test Framework (FTF)

AspectAssessment
MechanismBlueprint or C++ test actors placed in a dedicated map; driven by the Automation Tool via -ExecCmds "Automation RunAll" or -RunAutomationTests commandlet. Each functional test is a self-contained level actor with PrepareTest / IsReady / StartTest / OnTimeout lifecycle.
Blueprint supportNative. Test logic can live entirely in BP without any C++; aligns with D12 (BP-first). Individual tests can call gameplay functions, tick the game, and assert on state through the Automation Spec runner.
Screenshot comparisonBuilt-in via TakeAutomationScreenshot node (Blueprint) or AutomationCommon::TakeScreenshot (C++). Supports pixel-level comparison against a "ground truth" image baked into the project's Test/Screenshots directory. Result is pass/fail with a diff image written to the output directory.
Headless / CIRuns in -nullrhi mode for pure logic tests, but screenshot comparison requires a RHI-enabled (GPU) context. Must use -RHI=Vulkan on Linux. Works well with self-hosted GPU runners.
Cesium tile non-determinismTiles load asynchronously; FTF has no built-in "wait-for-tile-load" gate. A custom ATileLoadGate actor or UCesiumTileset::GetTotalNumberOfTilesCurrentlyNeeded() poll is required. See Determinism section below.
Timeline fitGood. No external tooling needed; entirely within UE5 project. Learning curve is low for Blueprint-first team. Test maps can be authored incrementally.
Solo operatorExcellent. No shared infrastructure; tests live in the project.

Candidate B — Gauntlet Automation Framework

AspectAssessment
MechanismExternal Python/C# controller launches a packaged game on a remote device, monitors it via telemetry, and validates exit code + metrics. Designed for large teams with device farms. Requires build packaging and device provisioning.
Blueprint supportIndirect. Gauntlet tests observe the game process; in-game logic still requires C++ event hooks or console command responders. Higher plumbing cost.
Timeline fitPoor for 4-week timeline. Setup cost of Gauntlet controller, device management, and metric collectors is non-trivial and mis-sized for a solo team.
Solo operatorOver-engineered. Gauntlet assumes device-farm topology.
VerdictNOT RECOMMENDED for this MVP. Revisit for v2.1+ if multi-platform device testing becomes a requirement.

Candidate C — PIE Screenshot Diff (Manual or Scripted)

AspectAssessment
MechanismPlay-In-Editor mode with -Game or packaged build launched with fixed render settings. Screenshot taken via console command HighResShot 1920x1080 or Shot. Diff done offline with Python (SSIM via scikit-image or ImageMagick).
IntegrationRequires an external Python wrapper script to orchestrate: launch game, set camera pose, wait for tile load, take shot, exit. Useful for the screenshot-diff baseline plan.
DeterminismLumen + screen-space effects are non-deterministic frame-to-frame. Fixed seed, time-of-day lock, TAA sample count, and tile-load gate are all required before the shot is taken. See Determinism section.
Timeline fitModerate. The harness is not complex but does require a Python wrapper and CI integration. Suitable as the screenshot-diff layer on top of FTF.

Candidate D — Playwright (v1 pattern extended)

Out of scope for v2 in-game tests. Playwright operates on a browser DOM; UE5 native builds expose no HTTP/DOM surface. Playwright remains authoritative for the v1 Three.js game (AC-V2-11) and must not be extended to drive UE5 tests. A thin Playwright wrapper that drives a locally-launched UE5 process via console sockets is architecturally possible but unjustified at this timeline.

Recommended Stack

Primary: UE5 Functional Test Framework (FTF) with Blueprint-authored tests in a dedicated test map (Levels/TestMap_QA). Covers functional ACs, state machine assertions, cannon fire rate, hit detection, crash detection, and georeferencing correctness.

Layer 2: PIE Screenshot Diff — packaged build launched with fixed camera poses, tile-load gate, time-of-day lock; Python wrapper performs SSIM comparison against frozen Google Earth references. Covers AC-V2-18 (photoreal screenshot-diff).

Layer 3: Python perf harness — launches packaged build, runs a 60-second flight loop above Inhauma, samples FPS via stat FPS console command redirected to a log file, validates 99th-percentile frame time. Covers AC-V2-17.

Layer 4: Playwright v1 suite — unchanged, run in parallel in its own CI lane. Covers AC-V2-20 (v1 suite green).

2. Test-Time Camera Setup

In the UE5 test, fixed camera poses are applied programmatically:

-- Blueprint pseudocode for each reference camera pose --

1. Set CesiumGeoreference.OriginLatitude  = -19.47
   Set CesiumGeoreference.OriginLongitude = -44.46
   Set CesiumGeoreference.OriginHeight    = 800.0

2. For each camera entry in cameras.json:
   a. Convert (lat, lon, alt) → UE world position via
      CesiumGeoreference.TransformLongitudeLatitudeHeightToUnreal()
   b. Set CameraActor.SetActorLocationAndRotation(uePos, ueRot)
   c. Disable all player pawns (no aircraft in shot — terrain only)
   d. Lock time-of-day: set CesiumSunSky.SolarTime = 12.0 (noon)
   e. Disable TAA temporal accumulation: r.TemporalAASamples 1
   f. Wait for tile-load gate (see Determinism section)
   g. Execute: HighResShot 1920x1080 Filename=cam-01-current.png
   h. Exit PIE / packaged game

3. Python post-process: compute SSIM(reference, current) per camera

3. Diff Algorithm and Thresholds

Recommended metric: SSIM (Structural Similarity Index Measure) as the primary gate, with perceptual hash (pHash) as a secondary early-rejection filter.
MetricLibraryThresholdRationale
pHash (early reject)imagehash (Python) Hamming distance ≤ 20 / 64 bits Fast O(1) filter. If pHash distance > 20, the images are structurally different (e.g., wrong location entirely). Run before SSIM to fail fast without GPU cost.
SSIM (primary gate)scikit-image.metrics.structural_similarity Fail if SSIM < 0.72 on any single camera
Fail if mean SSIM across all 4 cameras < 0.78		 SSIM 0.72 is intentionally permissive at the per-camera level to tolerate tile-streaming variation, Lumen GI indirect lighting differences, and sky colour shift from different GPU implementations. Mean 0.78 catches systematic drift. Thresholds are provisional and should be empirically calibrated from 3+ captures on the operator's dev machine before locking in SPEC.
Diff image outputGenerate diff-cam-01.png per camera showing pixel-wise delta heatmap Archived as CI artifact for operator eyeball review regardless of pass/fail.
Open Question OQ-01 status: The thresholds above (SSIM 0.72 / 0.78) are the game-tester's provisional proposal. The game-designer hat must validate that these thresholds do not permit visually unacceptable drift before SPEC approval (per Conflict Prediction C3 in the discovery report). If the game-designer objects with visual evidence, the synthesis stage must arbitrate.

4. Tolerance for Tile-Streaming Non-Determinism

This is the hardest problem in the screenshot-diff plan. Cesium tiles load asynchronously over HTTP. Without a tile-load gate, screenshots will contain partially loaded tiles (grey/black placeholders), producing SSIM values far below any reasonable threshold.

Recommended tile-load gate: Poll ACesium3DTileset::GetTotalNumberOfTilesCurrentlyNeeded() and ACesium3DTileset::GetTotalNumberOfTilesCurrentlyLoading(). Wait until tilesCurrentlyLoading == 0 && tilesNeeded > 0 for at least 2 consecutive frames (200ms). Additionally, enforce a hard maximum wait of 30 seconds from camera placement. If tiles do not fully load in 30 seconds, the test fails with TILE_LOAD_TIMEOUT rather than a false screenshot diff.

In offline tile-cache mode (dev workflow, cost control), all tiles should be cached locally and load in <2 seconds. The 30-second timeout is a CI safety net.

Determinism for Tile-Streaming Tests

Problem Statement

v1 (Three.js) achieves test determinism via ?testMode=1&seed=... URL parameters that freeze the PRNG and skip time-dependent animations. UE5 + Cesium does not have this mechanism by default. We need an equivalent.

Option A — testMode Console Variable

Recommended — direct analogue of v1's window.game.testMode.

Expose a UE5 console variable aero.testMode 1 (or a dedicated -aero_test_mode command-line argument) that, when set:

  • Locks time-of-day: CesiumSunSky.SolarTime = 12.0 (sun directly overhead, minimal shadow variation)
  • Disables TAA temporal accumulation: r.TemporalAASamples 1
  • Disables Lumen screen-space noise: r.Lumen.ScreenProbeGather.TemporalFilter 0
  • Disables motion blur: r.MotionBlur.Amount 0
  • Fixes camera at a documented pose (for screenshot tests)
  • Enables verbose tile-load telemetry logging
  • Shrinks service-scene animations (analogue of v1's 30-40s → 5s in testMode)
CriterionAssessment
CostLow. All console variables are already available in UE5.4. Blueprint-accessible via Execute Console Command node.
BenefitHigh. Eliminates lighting/shadow non-determinism between runs. Does not eliminate tile-streaming non-determinism but reduces it to the tile-load wait gate.
RiskDisabling TAA and Lumen temporal filter changes the visual output slightly versus production. Screenshot baselines must be captured with testMode enabled to avoid systematic SSIM bias.

Option B — Mock the Cesium Tile Request Layer (HTTP Intercept)

CriterionAssessment
MechanismRun a local HTTP proxy (e.g., mitmproxy or a simple Python http.server) that serves pre-cached tiles from disk instead of forwarding to the Google Tiles API.
BenefitEliminates both tile-streaming latency and API cost during CI tests. Every CI run uses exactly the same tiles, removing the SSIM variance from tile-version changes on Google's side.
CostMedium. Requires: (a) a tile snapshot for the 4 reference camera frustums, (b) a local proxy process started before the test, (c) overriding the Cesium tile server URL via GCP_TILE_PROXY_URL env var. Tile snapshots need to be Git LFS-stored (potentially large — 100–500 MB for 4 camera frusta at full LOD).
RiskTile caching must be refreshed when Google updates the Inhauma imagery (typically yearly). Stale tiles = stale baselines = false passes.
RecommendationPHASE 2 — implement for v2.0.1 after MVP proves the screenshot-diff harness works. Too much setup cost for 4-week timeline.

Option C — Pre-Cached Tile Snapshots in Test Data

CriterionAssessment
MechanismCesium for Unreal supports a "cache" directory (CesiumCache/) that stores downloaded tiles. Pre-populate this cache, commit to Git LFS, and configure the test runner to use it. The game then reads tiles from disk without any HTTP request.
BenefitZero API cost in CI; fully deterministic tile content.
CostThe Cesium disk cache format is not guaranteed stable across plugin versions. Cache invalidation on Cesium upgrade requires full re-capture. Storage size: 200–800 MB depending on frustum count.
RecommendationPHASE 2 — same as B. Evaluate after MVP.

MVP Determinism Recommendation

For MVP (4-week timeline): implement Option A (testMode console variable) for lighting/rendering determinism. Use the tile-load gate (poll tilesCurrentlyLoading == 0) for streaming determinism. Accept ~5-10 % SSIM variance across runs from tile-version differences; calibrate thresholds from 3+ sample runs before declaring baselines final.

Known UE5 Risks

Risk R-01 — Cesium for Unreal Version Ceiling at v2.21.1 for UE 5.4

Severity: HIGH
Research finding (May 2026): Cesium for Unreal v2.21.1 is the last release to support UE 5.4. All subsequent versions (v2.22.0+) require UE 5.5+. This creates an immediate risk: any Cesium bug discovered in v2.21.1 that is fixed in v2.22.0 cannot be backported to UE 5.4 without a manual patch. The project is pinned to a legacy Cesium version from day one.

Mitigation recommendations:

  • Evaluate upgrading to UE 5.5 before implementation starts (the latest Cesium versions would then be available). If D1 (UE 5.4 pin) is firm, accept the risk and document it in the SPEC risk register.
  • Pin exactly to CesiumForUnreal-v2.21.1 in the UE project's Plugins/ directory. Commit the plugin source or binary; do not rely on Marketplace auto-update.
  • Monitor CesiumGS/cesium-unreal issue tracker for bugs affecting v2.21.1 / UE 5.4 combinations — especially tile streaming and Linux Vulkan.

Risk R-02 — UE 5.4 + Lumen Screen-Space Artifacts in PIE and First Frames

Severity: MEDIUM
Lumen's screen-space global illumination and reflection captures are temporally accumulated. In PIE (Play-In-Editor) mode and in the first 2–3 seconds of a packaged game launch, Lumen GI is partially converged, producing visible dark splotches, bright halos, and flickering reflections. These are not bugs in the game — they are intended Lumen behaviour — but they will cause false screenshot-diff failures if screenshots are taken before convergence.

Mitigation:

  • In testMode, add a 2-second settle period after tile-load gate before taking any screenshot.
  • Consider disabling Lumen entirely in test map and using Baked Lighting for the 4 reference cameras — this eliminates temporal noise entirely at the cost of losing the real-time GI look. Screenshot baselines would then be captured with Baked Lighting too.
  • Alternative: set r.Lumen.ScreenProbeGather.TemporalFilter 0 in testMode to get a single-frame Lumen sample (noisier but deterministic per frame).

Risk R-03 — Functional Test Framework + Blueprint Nativization Quirks

Severity: MEDIUM
When Blueprint Nativization is enabled in Shipping builds (recommended in D12 for performance), the generated C++ nativized code can diverge from PIE Blueprint behaviour. Functional Tests that pass in PIE against non-nativized BP may fail in Shipping builds where the same logic runs as nativized C++. Known edge cases: floating-point comparison rounding differences, event dispatch order, and null-check semantics.

Mitigation:

  • Run the FTF suite in both Development build (non-nativized) and Shipping build (nativized) before tagging a release.
  • Avoid relying on exact floating-point equality in FTF assertions; use tolerances (e.g., SSIM ≥ 0.72, not == 0.847231).
  • Gate AC-V2-05 (cannon fire rate) with a jitter tolerance (±5 ms inter-spawn interval) rather than an exact count, since nativization may reorder event dispatch by 1 tick.

Risk R-04 — Headless CI Rendering on Linux (No GPU on GitHub-Hosted Runners)

Severity: MEDIUM-HIGH for screenshot tests
GitHub-hosted Linux runners do not have a physical GPU. UE5 screenshot tests and perf tests require GPU rendering (Vulkan on Linux). The options are: (a) self-hosted GPU runner (operator's machine), (b) cloud GPU VM (Azure NV-series or AWS g4dn), or (c) limit screenshot tests to Windows CI only.

Mitigation (input to DevOps — see DevOps Inputs section):

  • Recommend self-hosted GitHub Actions runner on operator's dev machine (RTX 3060) for screenshot-diff and perf tests. This satisfies both the GPU requirement and the AC-V2-17 "RTX 3060" perf gate simultaneously.
  • Functional tests that do not require GPU (state machine, fire rate, hit detection) CAN run headless via -nullrhi on standard GitHub-hosted runners.

Risk R-05 — Cesium World Bounds Checks Conflicting with UE5 Origin Rebasing

Severity: MEDIUM
UE5's "Enable World Bounds Checks" will attempt to teleport any pawn that strays far from the origin back toward origin — conflicting with Cesium's coordinate system where world positions corresponding to real-world coordinates can be very large numbers. This must be disabled. If not disabled, the aircraft will be teleported unexpectedly when flying at the edges of the 20 km radius, causing false crash detections.

Mitigation:

  • Add an FTF assertion: WorldSettings.bEnableWorldBoundsChecks == false.
  • Document in the SPEC "Platform Prerequisites" section.

Risk R-06 — Cesium Tile Streaming Cost Spike During CI Screenshot Tests

Severity: MEDIUM
Each CI screenshot-diff run that uses live Google Maps Tiles API will stream tiles for 4 camera frusta. At high LOD, this may consume significant API quota. With a hard cap of USD 20/month, an unguarded CI run frequency (e.g., every PR) could exhaust the budget within days.

Mitigation:

  • Screenshot-diff tests run only on tag CI (not on every PR). PR CI uses -nullrhi functional tests only.
  • Implement an offline tile-cache warm-up (Option C from Determinism section) for Phase 2. For MVP: accept live tile costs on tag CI only (estimated < 5 tag runs per month).

Risk R-07 — Google Earth Imagery License for CI Baselines

Severity: LOW for private repo, MEDIUM if ever published
Google Earth imagery screenshots are covered by Google's ToS for personal non-commercial use. Storing them in a private CI repo as automation baselines is a grey area. Google explicitly permits "printing" screenshots but restricts redistribution.

Mitigation:

  • Keep reference screenshots in a private repository (not in the public tauan-games repo if it becomes public).
  • Alternative source: Bing Maps tiles (permissive license for derivative works) or OpenAerialMap (CC-BY). These cover the Inhauma area adequately. Validate image quality before committing to an alternative source.

Inputs to DevOps

The following items are informational requirements that the devops-engineer must accommodate. These are not objections from the game-tester role — they are constraints the test infrastructure must satisfy.

Input DevOps-1 — Self-hosted GPU runner required for screenshot-diff and perf tests.
AC-V2-17 (60 FPS @ RTX 3060) and AC-V2-18 (screenshot-diff) require a physical GPU. GitHub-hosted runners are GPU-less on Linux. Recommendation: register the operator's dev machine (RTX 3060, Windows or Linux) as a self-hosted GitHub Actions runner. Label: gpu-rtx3060. Tag-only CI jobs (screenshot-diff, perf) must specify runs-on: [self-hosted, gpu-rtx3060].
Input DevOps-2 — Google Maps Tiles API secret in CI.
The screenshot-diff test (AC-V2-18) and tile-load gate test (AC-V2-13) require the GCP Tile API key in CI. Store as GitHub Actions secret GCP_TILE_API_KEY (injected as env var). Do NOT expose in PR-triggered workflows (only tag-triggered). The key is the same one documented in D3 / OQ-03.
Input DevOps-3 — Git LFS for reference screenshots.
Approximately 20 MB of PNG reference images (4 cameras × ~5 MB) will be stored in tests/aero-fighters-v2/screenshot-diff/references/ via Git LFS. The CI runner must have LFS configured. GitHub Actions LFS support is built-in; verify the self-hosted runner also has git-lfs installed.
Input DevOps-4 — CI matrix: two parallel lanes (v1 Playwright + v2 UE5).
Per D9 (parallel v1 evolution), the v1 Playwright suite must run in a separate CI lane from the v2 UE5 tests. Both must complete on the same tag CI run. Suggested GitHub Actions structure: two jobs in the same workflow — v1-playwright (GitHub-hosted Linux, Chromium) and v2-ue5 (self-hosted GPU runner). Tag naming convention must distinguish v1 and v2: aero-v1-X.Y.Z for v1 releases, aero-v2-X.Y.Z for v2.
Input DevOps-5 — Python test harness dependencies.
The screenshot-diff and perf harness require Python 3.10+ with packages: scikit-image (SSIM), imagehash (pHash), Pillow, numpy. These should be installed on the self-hosted GPU runner or via a pip install -r requirements-test.txt step in CI.
Input DevOps-6 — UE5 commandlet for non-GPU FTF.
Functional tests that don't need GPU (state machine, fire rate, georef assertions) must run via UE5 automation commandlet with -nullrhi -unattended flags. The exact invocation is: UE4Editor-Cmd.exe AeroFightersV2 -ExecCmds="Automation RunAll;Quit" -unattended -nullrhi -log -logfile=ftf-results.txt (Windows) or equivalent on Linux. DevOps must configure this step in the non-GPU CI job. FTF results file must be parsed and test counts validated.
