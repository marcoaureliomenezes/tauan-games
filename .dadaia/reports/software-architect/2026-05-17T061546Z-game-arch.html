This report consumes the 15 locked decisions, 13 deferred questions, and 7 conflict predictions from the discovery report; cross-references the v1 Three.js architecture (aero-fighters/ARCHITECTURE.md, CONVENTIONS.md, and the actual src tree at aero-fighters/src/); and applies the architect-design-patterns, architect-code-audit, game-platform-unreal, game-unreal-developer, and game-flight-dynamics skill protocols. Sections are ordered per the workflow contract.

Top-3 architectural risks surfaced (full detail in Cesium + UE 5.4 Risks and the timeline section):

  1. UE 5.4 pin vs Cesium for Unreal release cadence. Cesium ships frequently and occasionally drops older UE versions on major bumps; we cannot upgrade UE inside MVP.
  2. Cesium georef coordinate-system leak. If gameplay code (AA gun position, weapon hitscan, future JSBSim hook) addresses world in raw FVector cm without going through a georef-aware seam, the codebase becomes un-portable to any other map and impossible to swap Cesium for a fallback.
  3. BP-first + 60 FPS floor + Lumen + Cesium tile streaming = no headroom. The per-frame budget on a 3060 at 1080p is ~16.6 ms; Lumen + Cesium tile decode + Niagara already plausibly consume 12–14 ms before a single gameplay tick. The flight model cannot live in Blueprint Tick at MVP scale without measurable risk.

Top-3 design-pattern commitments (full detail in Pattern Fit):

  1. Component pattern via UE Actor-Component for flight, weapons, health, damage — already idiomatic to UE5 and matches v1's module decomposition.
  2. Hexagonal seam (Port & Adapter) at the georef boundary — domain code consumes a UWorldGeoreferenceComponent interface; the Cesium adapter is the only thing that knows Cesium types. Non-negotiable.
  3. Observer (UE Delegates) for cross-actor events — enemy destroyed, projectile hit, sortie-state transitions — instead of singleton GameMode pull or Cast-To chains.

Why seven, not one monolith and not twenty? Seven is the smallest split that isolates Cesium in exactly one module (most important architectural decision in the whole project) and keeps the rebuild cost of a BP-only iteration under one minute. More modules would multiply .Build.cs maintenance with no parallel-dev benefit since the team is solo (D13). Fewer modules force Cesium types into Gameplay headers, which prevents ever swapping Cesium for a Google Tiles SDK fallback or a baked offline tileset later.

CRITICAL Cesium must not leak past AeroFightersGeoref

Location: Module dependency graph
Issue: If any other module's public header includes a Cesium type, the entire project transitively depends on Cesium. This kills the if-slip rule (D17) — there is no recovery path that drops the Linux build or swaps to a baked tileset without a multi-week rewrite.
Why it matters: Cesium for Unreal averages a release every 4–6 weeks and has historically dropped UE 5.0/5.1/5.2 on major bumps. UE 5.5 is already out; UE 5.6 ships during our MVP window. If Cesium 3.x drops 5.4 mid-MVP, we either freeze on a Cesium release (no bugfixes, no Linux improvements) or rebase the entire engine version mid-sprint — neither is acceptable in a 4-week timeline.
Trade-off if fixed: The port abstraction costs one C++ interface (IWorldGeoreferenceProvider with ~6 methods) and one adapter. The benefit is total: Cesium can be deleted and replaced in days, not weeks.
Recommendation: Add a static .Build.cs lint to AeroFightersHarness that fails CI if #include "Cesium appears in any header under AeroFightersCore, AeroFightersGameplay, AeroFightersCombat, AeroFightersHUD, AeroFightersAudio.

Layer rules (enforced by Build.cs PublicDependencyModuleNames):

  1. Core imports nothing from this project. No UE world types (UWorld, FVector world-space) in public headers.
  2. Georef imports Core only. Cesium types live only in .cpp files and module-private headers.
  3. Gameplay imports Core + Georef. Never Combat, HUD, Audio, Harness.
  4. Combat imports Gameplay (Pawn base) + Core. Never Georef directly — combat asks the FlightComponent for FGeoCoord when needed.
  5. HUD and Audio import Gameplay read-only (subscribe to delegates; never mutate).
  6. Harness imports anything (EditorOnly module, stripped in Shipping).

BP-to-C++ promotion trigger (closes OQ-02)

The trigger has to be measurable on Day 1, not eyeballed at week 3. Proposal:

Promotion trigger (proposed for synthesis):
  1. Hard rule: any class whose Tick consumes more than 0.30 ms / frame on the target hardware (3060 @ 1080p) as measured by stat startfile + Insights for 30 s of representative gameplay must be promoted to C++ before the next merge.
  2. Soft rule: any class whose BP graph exceeds 150 nodes or whose function call depth exceeds 5 levels must be reviewed for promotion at the end of every week (game-developer's call per Decision Matrix).
  3. Aggregate budget: total BP VM cost reported by Insights must stay under 2.0 ms / frame. If it crosses 2.0 ms, the largest-cost BP class is promoted that week. This protects the 16.6 ms frame budget proactively.
The 0.30 ms threshold is derived as follows: total budget 16.6 ms; Lumen + Cesium streaming reasonably claim 10–12 ms; rendering misc 2 ms; we have ~3 ms for all CPU gameplay. With ~10 BP classes ticking, 0.30 ms per class consumes the whole budget — so 0.30 ms is the per-class fairness ceiling.

HIGH BP-first is not safe for the flight tick path

Location: UFlightArcadeComponent::TickComponent (proposed)
Issue: v1's updatePlayer() is 75 LOC of quaternion math, throttle convergence, stall checks, and ground-contact lookup. In Blueprint, the same logic becomes ~150 nodes and runs through the VM at roughly 5–15× the cost of native code. At 60 FPS that's potentially 1.5–4 ms just for the player tick — half our entire CPU budget.
Why it matters: If we start in BP and discover the perf problem in week 3, we have to rewrite the flight model AND retune every coefficient because BP-vs-C++ rounding behavior diverges slightly. This is exactly the kind of mid-sprint surprise the discovery report warned about (Tension T2).
Trade-off: Starting in C++ costs ~2 days of upfront work but eliminates a class of week-3 surprise. The BP-first policy still applies to gameplay rules that designers want to tweak — flight feel is one of those, but the math underneath is not.
Recommendation: Expose tuning in BP via UPROPERTY(EditAnywhere, BlueprintReadWrite, Category="Flight") on the C++ component (throttle rate, pitch rate, max speed, stall threshold), so the operator and game-developer iterate values in editor without touching code. This satisfies both D12 (BP-first iteration) and the perf constraint. Per the Decision Matrix, this is an objection — game-developer has tie-breaker on UE5-specific idioms; this report recommends, the synthesis stage decides with game-developer's input.

State Machine Reuse

v1 has an explicit sortie state machine in aero-fighters/src/sortie-state.js: 14 states (MENU, TAXI_OUT, TAKEOFF_ROLL, AIRBORNE, MISSION_ACTIVE, RETURN_TO_BASE, LANDING_ROLL, TAXI_IN, SERVICE_SCENE, NEXT_SORTIE_READY, MAYDAY, EJECTION, CRASHED, MISSION_FAILED) wired by a TRANSITIONS map keyed by state:event.

The MVP scope per D8 is: flight + cannon + one enemy AA gun. No takeoff/landing realism, no service scene, no ejection, no parachute, no mission completion loop. The relevant question is: does v2 MVP need any SM at all?

v1 stateNeeded in v2 MVP?Why
MENUYesInitial state before "Press to start". Cesium tile pre-warm happens here.
TAXI_OUT / TAKEOFF_ROLL / LANDING_ROLL / TAXI_INNoD4 + D8: arcade flight only, no airport realism in MVP.
AIRBORNEYesDefault flight state.
MISSION_ACTIVE / RETURN_TO_BASENoNo mission completion loop in MVP.
SERVICE_SCENE / NEXT_SORTIE_READYNoOut of MVP scope (mission realism is v2.1+).
MAYDAY / EJECTIONNoOut of MVP scope.
CRASHEDYesPlayer aircraft can hit terrain (Inhaúma has hills); this must be representable.
MISSION_FAILEDMaybeEquivalent to CRASHED in MVP — fold in.

Recommendation: REDESIGNED (slimmed) — not omitted

Proposal: Port the SM structure from v1 — keep the same state/event/transition pattern, USTRUCT-encoded — but seed it with only the four states MVP needs: 
UENUM(BlueprintType)
enum class ESortieState : uint8
{
    Menu,
    Airborne,
    Crashed,
    SessionEnded
};

UENUM(BlueprintType)
enum class ESortieEvent : uint8
{
    StartFlight,        // Menu  → Airborne
    TerrainImpact,      // Airborne → Crashed
    Respawn,            // Crashed → Airborne
    QuitToMenu          // *      → SessionEnded
};

USTRUCT(BlueprintType)
struct FSortieMachine
{
    GENERATED_BODY()
    UPROPERTY(BlueprintReadOnly) ESortieState State = ESortieState::Menu;
    UPROPERTY(BlueprintReadOnly) TArray<FSortieTransitionRecord> History;
};

Why redesigned, not omitted:

  • An explicit SM is the cheapest way to keep the "flight + cannon + AA" loop testable without booleans-on-booleans (v1's lesson, per mission-realism SPEC §5 FR-01).
  • v2.1 (full arcade combat) will need the v1 SM back. Designing the slim version with the v1 shape means v2.1 only adds states/events to existing data, never restructures.
  • The slim version is ~40 LOC of C++ in AeroFightersCore. Cost is trivial; the testability win is large because the SM is engine-free and headlessly testable.

MEDIUM Do not copy v1's TRANSITIONS Map verbatim into C++

Location: aero-fighters/src/sortie-state.js lines 37–52
Issue: v1 uses a Map keyed by template-string concatenation (${state}:${event}). Idiomatic for JS, anti-idiomatic for UE5 C++ where the equivalent is a switch on a tuple or a TMap<FStateEventKey, ESortieState>. A naive port produces an unidiomatic, hard-to-debug structure.
Why it matters: The state machine is a documentation artifact as much as it is code — game-developer and game-tester will both read it. A non-idiomatic structure adds friction to every change.
Trade-off: Using a switch is more verbose but auto-completes, fails compile on typos, and shows up cleanly in stack traces. Worth the verbosity.
Recommendation: Implement transitions as a single FSortieMachine::Transition(ESortieEvent) method with a switch on (State, Event) tuples. v1's history ring-buffer (line 70: history.length > 32) is a good idea — port it as TRingBuffer<FSortieTransitionRecord, 32>.

Cesium + UE 5.4 Risks

The 4-week timeline plus the UE 5.4 hard pin plus the Cesium plugin dependency create a small number of high-blast-radius risks. Each is enumerated with a concrete mitigation that the synthesis stage can fold into SPEC §Risk Register.

CRITICAL R1 — Cesium for Unreal drops UE 5.4 mid-MVP

Trigger: Cesium ships a major version (e.g. 3.x → 4.x) that drops 5.4 support because the team moved to 5.5/5.6. Historically Cesium does this every ~12 months.
Blast radius: Cannot pull bugfixes; cannot pull new tile-streaming optimizations (directly affects 60 FPS gate on 3060); cannot follow the Cesium Linux support timeline.
Mitigation:
  1. Pin Cesium plugin to an exact commit SHA via git submodule (not a release tag — tags can move). The pin lives in aero-fighters-v2/Plugins/CesiumForUnreal/.commit-sha.
  2. Mirror the pinned source to github.com/<operator>/cesium-unreal-pin so the upstream cannot delete it.
  3. The if-slip rule (D17) must include "Cesium-pin retreat" as an explicit option: if a Cesium bug is found mid-sprint and the bugfix only ships in a newer Cesium that drops 5.4, freeze on the buggy version and document the workaround for the bug.
Owner: devops-engineer (CI/build per Matrix) — this is an objection passed to their report.

CRITICAL R2 — Cesium georef seam violation

Trigger: Any gameplay code addresses world position in raw FVector centimeters (UE convention) instead of FGeoCoord. As soon as one such line exists in Combat or Gameplay, the codebase ties forever to "the current CesiumGeoreference origin" and silently breaks if the origin moves (a thing ACesiumGeoreference can do via SubLevels).
Blast radius: AA gun placement breaks; projectile hitscan breaks for tiles streaming in at the world edge; future map (Inhaúma 40 km radius, second map) requires rewriting every cm-coordinate literal.
Mitigation:
  1. All AA-gun and any future placement use FGeoCoord (lat/lon/alt) serialized in a DataAsset. The UWorldGeoreferenceComponent::SpawnAtGeoCoord() call is the only way to put something in the world.
  2. Add a CI lint that flags magic-number FVector( in gameplay .cpp files (whitelist for clearly-relative offsets).
  3. The screenshot-diff harness (Harness module) uses FGeoCoord for its fixed poses (per AC-V2-02 / OQ-07 deterministic captures).

HIGH R3 — Cesium tile-streaming I/O on the game thread

Trigger: Cesium's tile loader is async but its decode/upload occasionally hitches on the game thread (well-known issue tracked in Cesium GitHub). On a 3060 with thermal throttling, this can produce 50–80 ms spikes that violate AC-V2-07's 99th-percentile ≤ 18.5 ms.
Blast radius: Perf-gate test in CI fails intermittently; player sees micro-stutter during fast traversal.
Mitigation:
  1. Pre-warm tiles for the MVP play area at level-load (block in MENU state on a loader, not in AIRBORNE).
  2. Cap MaxTileCacheBytes to a value derived from RTX 3060's 12 GB VRAM minus Lumen/Niagara budgets — propose 3 GB as starting point; tune in PLAN.
  3. Profile with Unreal Insights every PR; the perf gate (AC-V2-07) is the canary.

HIGH R4 — Linux Vulkan vs Cesium tile decode quirks

Trigger: Cesium for Unreal's Linux support lags Windows by ~one minor version. Shader compile differences between Vulkan (Linux) and DX12 (Windows) produce different screenshot-diff outputs at the same geo-pose — failing AC-V2-06 on Linux only.
Blast radius: Linux build (AC-V2-09) cannot pass photoreal acceptance (AC-V2-06). This is exactly the condition that fires the if-slip rule (D17).
Mitigation:
  1. Maintain two reference-image baselines, one per platform, from week 1. Diff is per-platform, not cross-platform.
  2. Architecture decision (objection to game-tester): the screenshot-diff tolerance metric must be SSIM or perceptual-hash, NOT per-pixel — per-pixel will fail across shader variants. (Matrix: game-tester has tie-breaker; this is the architect's evidence-backed recommendation.)
  3. If R4 fires at week 3, the if-slip rule drops Linux. The georef seam (R2 mitigation) means this is a Build.cs change, not a code rewrite.

MEDIUM R5 — Cesium plugin update changes FGeoreferenceOriginPlacement default

Trigger: Cesium has changed the default origin placement (CartographicOrigin vs TrueOrigin) across minor versions in the past. Any unpinned default breaks FGeoCoord ↔ FVector round-trips silently.
Mitigation:
  1. The UWorldGeoreferenceComponent sets all Cesium config explicitly in code, never relies on defaults.
  2. Round-trip assertion in the Harness: place at Inhaúma origin, get FGeoCoord, convert back, verify cm-equality. Runs on every CI commit.

MEDIUM R6 — Google 3D Tiles license terms vs offline cache (OQ-05)

Trigger: Google's 3D Tiles ToS restricts caching duration/scope. The dev offline cache (D3) might violate cache-lifetime terms; CI screenshot-diff baselines (OQ-06) might violate reference-imagery use.
Mitigation: This is a devops-engineer + product-engineer concern, surfaced here because the architecture (offline cache in repo, baselines in repo) is what triggers it. The Harness module must not commit tile bytes into git — it points at a local cache path declared by env var.

MEDIUM R7 — UE 5.4 itself is on a maintenance branch

Trigger: Epic ships UE 5.5 (already shipped) and 5.6; 5.4 receives only critical patches by end of 2026. Any UE-side bug we hit on 5.4 may not be fixed upstream.
Mitigation:
  1. Cross-reference any UE bug we hit against the 5.5 changelog before reporting; the fix may exist there.
  2. D1 is locked, so a UE bump is out of MVP scope. v2.1 should plan a UE-version review in its discovery.

Pattern Fit

Applied the architect-design-patterns evaluation protocol to every pattern the proposed architecture either commits to or deliberately rejects.

Patterns committed to (CORRECT)

Component (UE Actor-Component) UFlightArcadeComponent, UCannonComponent, UEngineAudioComponent, UWorldGeoreferenceComponent Idiomatic to UE5; matches v1's "entities" module decomposition (player/targets/ bullets/missiles); cleanly maps gameplay capabilities to composable pieces.

Patterns deliberately deferred or rejected

PatternStatusWhy
Strategy for swappable flight model (arcade ↔ JSBSim) DEFERRED to v2.1 D4 explicitly defers JSBSim. Designing the Strategy now is Speculative Generality (anti-pattern) with zero current use cases. The Component split (UFlightArcadeComponent) leaves room for UFlightJSBSimComponent to plug in via the same Pawn slot when needed.

Inheritance for behavior variation REJECTED If v2.1 adds enemy types, they're UActorComponent compositions on a generic AEnemyPawn, not subclasses. v1's lesson: makeBase/makeFactory/makeBuilding/makeConvoy/makeAAGun were procedures not subclasses, and that was the right call.

Anti-patterns to actively prevent

HIGH Anti-pattern watch: "God GameMode"

Risk: UE encourages putting everything in AGameMode. Easy slope: spawn rules, score, transition logic, projectile pool init, HUD wiring all end up there. v1's equivalent was main.js growing to 449 LOC; the proposed v2 layout keeps AAeroGameMode as a thin orchestrator (~120 LOC C++ + BP rules) that delegates to subsystems and components.
Hard rule for MVP: AAeroGameMode ≤ 150 LOC C++ and ≤ 80 BP nodes. Above that, fields move out into a Subsystem or Component.

MEDIUM Anti-pattern watch: Cast-To chains

Risk: BP-first invites Cast To AFighterPawn → Cast To UFlightArcadeComponent → ... chains as the easy way to wire UI. Each Cast-To loads the entire class asset, slowing PIE startup and tying assets together at the asset-reference level.
Mitigation: Define BP Interfaces (BPI_FlightTelemetry, BPI_Damageable) and have HUD widgets consume the interface. Documented in the game-unreal-developer skill (Cast To "com moderação — prefira Interfaces").

MEDIUM Anti-pattern watch: Anemic Domain Model in Core

Risk: FGeoCoord and FSortieMachine become data bags that Gameplay reaches into and mutates from the outside.
Mitigation: Both types expose only behavior methods (FSortieMachine::Transition, FGeoCoord::Distance) — public fields are read-only. UDamageModel exposes ApplyDamage(amount, source), not raw HP mutation.

4-Week Timeline Architectural Concessions

Discovery report tags D10 as RISK and explicitly asks me to opine on whether the 4-week timeline is architecturally feasible given D6 (60 FPS on 3060). My assessment: feasible only if the three concessions below are accepted and the one hard line is held.

Acceptable concessions (architecturally justified shortcuts)

JUSTIFIED C1 — Skip the AeroFightersAudio module for MVP

Concession: Roll audio into AeroFightersGameplay as a single UEngineAudioComponent. Refactor out to its own module in v2.1 if/when sound design grows beyond engine/cannon/explosion.
Why justified: v1 audio is 345 LOC; for MVP we need ~3 sounds (engine RPM, cannon, explosion). One Component on the Pawn covers it. Building a separate module costs a half-day of .Build.cs wiring with no architectural payoff at MVP scale.
Reversibility: Trivial. Moving a Component to its own module is a file-move + Build.cs edit; no API changes.

JUSTIFIED C2 — Skip the AeroFightersHUD module for MVP

Concession: UMG widgets and the AAeroHUD class live in AeroFightersGameplay. Pull out to its own module in v2.1 if HUD complexity grows (multiple HUD modes, replay overlay, etc.).
Why justified: MVP HUD = airspeed + altitude + crosshair + RTB marker. Four widgets, ~100 LOC of UMG. Module boundary cost (Build.cs, header export discipline) buys nothing at this size.
Reversibility: Trivial.

JUSTIFIED C3 — One .uproject + one .uplugin only (Cesium)

Concession: Do not create custom plugins for Core / Georef / Gameplay etc. They are all game modules in the same .uproject, not plugins.
Why justified: Custom UE plugins exist to share code across multiple projects. We have one project. Plugin overhead (separate Build.cs, plugin descriptor, hot-reload quirks) buys nothing.
Reversibility: Easy. Game module → plugin migration is a known UE refactor.

JUSTIFIED C4 — Defer the Strategy pattern around flight model

Concession: No IFlightModel interface. AFighterPawn directly owns UFlightArcadeComponent. JSBSim swap is a future PR, not a day-1 abstraction.
Why justified: Speculative Generality (per design-patterns skill, MEDIUM anti-pattern). v2.1 will be the second use case that justifies the interface — until then it is over-engineering.
Reversibility: Easy. Extract-interface refactor on one component is a known move.

Hard lines (concessions that look tempting but are forbidden)

FORBIDDEN F1 — Do not skip the georef abstraction

Tempting shortcut: "We're solo, we know it's only Cesium, just type FVector directly and ship."
Why forbidden: Every cm-literal in gameplay code is a future migration cost. The if-slip rule (D17) becomes un-executable. The AA gun placement (AC-V2-05) bakes into Cesium's CesiumGeoreference origin choice and silently breaks if that origin moves. This is exactly the "build on stale layers" pattern the architect-code-audit skill flags as the primary source of catastrophic incidents.
What to do instead: Spend the 4 hours to define IWorldGeoreferenceProvider + UCesiumGeorefAdapter on day 1. That's it.

FORBIDDEN F2 — Do not put flight math in Blueprint

Tempting shortcut: "D12 says BP-first, so let's BP the FlightComponent and profile later."
Why forbidden: See HIGH finding in "C++/Blueprint Boundary" — the rewrite cost discovered at week 3 plus the retune cost is a guaranteed timeline slip.
What to do instead: C++ UFlightArcadeComponent from day 1 with all coefficients as BlueprintReadWrite UPROPERTY. Designers still iterate in editor — that's the spirit of BP-first satisfied without paying the perf bill. (Recall: per Matrix, game-developer has tie-breaker on this; this report is objection-with-evidence, not a unilateral decision.)

FORBIDDEN F3 — Do not commit Cesium tile bytes to git

Tempting shortcut: "Cache tiles in the repo for offline dev workflow (D3)."
Why forbidden: Google's 3D Tiles ToS restricts persistent caching; repo blows up to GBs (architectural impact: every CI clone is slow); license risk (R6).
What to do instead: The Harness module reads a cache path from an env var (AERO_TILE_CACHE_DIR). The dev cache lives under the user's home; CI re-fetches on cold runs with the spend cap as the safety net (D3, AC-V2-10). Synthesis should hand this off to devops-engineer.

FORBIDDEN F4 — Do not let Combat reach into Gameplay GameState mutably

Tempting shortcut: "Combat code does GameState->Score += 100 directly when an enemy dies."
Why forbidden: v1's lesson exactly — spawnBullet wrote to game.projectiles, killTarget wrote to game.score, side-effects everywhere. ARCHITECTURE.md flagged this as smell #2 in v1.
What to do instead: Combat broadcasts FOnEnemyDestroyed.Broadcast(points). GameState subscribes. One writer per field; Observer pattern as documented above.
