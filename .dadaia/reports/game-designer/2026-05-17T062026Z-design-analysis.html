Map Feasibility

Required section per workflow contract — covers tile budget, LOD strategy, boundary behavior, and CesiumGeoreference origin validation.

1. Google Maps Tile API Budget Analysis at 20 km Box

Pricing Research Summary

Authoritative pricing for the Map Tiles API / Photorealistic 3D Tiles SKU was not publicly enumerated on standard pricing pages at time of research (2026-05-17). Based on research conducted via developers.google.com/maps/documentation/tile/usage-and-billing and mapsplatform.google.com/pricing, the following facts are confirmed:

  • Root tile requests are billable under the Photorealistic 3D Tiles SKU.
  • Session token requests, viewport information requests, and child tile requests (tiles fetched after the root session) are classified as non-quota-impacting but may still be billable depending on tier agreement.
  • Google offers a $200/month promotional credit applying to qualifying SKUs (confirmed active through at least early 2025; operator must verify current status against their GCP project).
  • The Enterprise subscription tier is required for the Photorealistic 3D Tiles product, which may imply a minimum spend floor above the $20 hard cap. This is the single most urgent open question before any implementation begins.
  • Attribution must display the Google Maps logo (min 16dp height) plus per-tile data-provider credits aggregated along the bottom of the render viewport. Cesium for Unreal exposes ShowCreditsOnScreen which must be true at all times the map is visible.
CRITICAL BUDGET RISK: If Photorealistic 3D Tiles requires the Enterprise tier, the hard cap of USD 20/month may be structurally unachievable through the standard self-serve GCP console. The operator must verify: (1) whether their personal GCP project has Photorealistic 3D Tiles enabled at pay-as-you-go rates, and (2) whether a $200/month promotional credit still applies to their account, which would more than cover the $20 cap. If Enterprise is mandatory, the fallback (Cesium World Terrain + Bing Aerial) must become the primary path, not the fallback.

Per-Session Tile Request Estimate

Assumptions:
Region: 20 km radius = 40 km x 40 km box = 1,600 km²
Tile zoom level 15 (approx. 1,000m resolution) → roughly 64 tiles cover 1,600 km²
Tile zoom level 16 (approx. 500m resolution) → roughly 256 tiles cover 1,600 km²
Tile zoom level 17 (approx. 250m resolution) → roughly 1,024 tiles cover 1,600 km²

Cesium streaming model:
Cesium does NOT fetch all tiles upfront. It streams only tiles in frustum + LOD budget.
At any frame: ~20–60 visible tiles (frustum angle ~60°, LOD depth 3–5 levels).
A 30-minute flight session fetches new tiles as the camera moves.
Conservative estimate: 500–1,500 unique tile fetches per 30-minute session.

Cost estimate (if billed at ~$0.004/request, common Maps API rate):
Per session: 1,000 requests × $0.004 = $0.004–$0.006 per session
Sessions to hit $20 cap at $0.005/session: 4,000 sessions/month
Dev + CI sessions budgeted: ~200 live-tile sessions/month (with offline cache for remainder)
→ Estimated cost at this rate: ~$1.00/month — well within $20 cap

WARNING: If billed per-byte (some GCP tile products): a 3D tile can be 5–500 KB. At 1,000 tiles × 200 KB avg × $0.008/MB = ~$1.60/session — much higher. The operator must check their GCP billing dashboard after first real session.
Design recommendation: Implement the offline tile cache as the default dev workflow from day one. Even if costs are low, the cache protects against rate spikes during rapid iteration. A populated 20 km box cache at LOD 14-16 fits in approximately 2–8 GB on disk — a reasonable trade for budget protection.

GCP Billing Alert Configuration (Design Requirement)

Per AC-V2-10, GCP billing alerts must be configured at 50%, 80%, and 100% of the $20 cap before the first live tile session. This is a hard prerequisite, not optional polish. The design places one AA gun and the aircraft spawn point over Inhaúma; any screenshot-diff test that loads live tiles also draws from this budget.

2. LOD Strategy

ParameterRecommended ValueRationale
Maximum Screen Space Error (SSE) 16 px Cesium for Unreal default is 16 px. Lowering to 8 px doubles tile requests without perceptible quality gain at typical flight altitude (200–800m AGL). Keep at 16 for MVP budget control.
Target view distance 20 km (matching playable box) Cesium's LOD system auto-culls beyond the horizon. 20 km at 800m altitude provides ~160 km visual horizon (Earth curvature) but tile fetch is bounded by frustum + SSE, not raw distance.
Prefetch radius 2 km ahead of aircraft At max speed (v1 arcade: 80 m/s ≈ 288 km/h), 2 km ahead = ~7 seconds of prefetch buffer. Prevents visible tile pop-in during straight-line flight.
Culling below terrain Enabled (Cesium default) Tiles occluded by local terrain should not be fetched. Reduces budget by 20–40% in mountainous sub-regions near Inhaúma (the Sete Lagoas highlands to the northeast).
Maximum simultaneous tile requests 18 (Cesium default) Adequate for RTX 3060 + 12 GB VRAM. Do not increase in MVP — it raises peak API request bursts and potential costs.

3. Fallback Strategy: Cesium World Terrain + Bing Maps Aerial

The operator confirmed this as the fallback plan. Design analysis supports and refines it:

  • Cesium World Terrain: Free under Cesium ion account (up to quota). Provides accurate DEM (Digital Elevation Model) for the Inhaúma region. Elevation data for lat -19.47, lon -44.46 shows mild topography (200–900m ASL) typical of the Minas Gerais highlands — adequately covered by CWT.
  • Bing Maps Aerial: Cesium for Unreal ships a BingMapsRasterOverlay. Bing imagery for the Inhaúma area is available but resolution is ~0.5–1m/px (vs Google's ~0.1–0.3m/px for photoreal). The visual gap is significant but the game remains geographically grounded.
  • Fallback trigger: If Google tiles fail to load (API auth error, budget exceeded, network outage), Cesium for Unreal should fall back to CWT + Bing automatically if configured as secondary overlay in the 3D Tileset stack. The ion_access_token environment variable controls this split.
  • Design impact: Under the fallback, the photoreal acceptance criterion (AC-V2-06 screenshot-diff) cannot pass. The fallback mode is a "gameplay continuity" mode, not a "release" mode. SPEC should encode this explicitly as a degraded mode flag.

4. Boundary Behavior: What Happens at the Edge of the 20 km Box?

The discovery report does not specify boundary behavior. This section is the game-designer's design decision.

Design decision (hard soft-wall with visual fade):
  • At 18 km from origin, a visible "Restricted Zone" warning appears in the HUD.
  • At 19.5 km, atmospheric fog density increases sharply (UE5 Exponential Height Fog, density factor ×10 over 500m). The terrain fades to white-grey haze.
  • At 20 km, an invisible forcefield (UE5 physics volume or Blueprint spring force) gently pushes the aircraft back toward center. The force is proportional to distance past the boundary — not a wall, but a strong wind.
  • No "game over" for boundary breach in MVP. The boundary is advisory, not lethal. This matches the arcade-combat identity (D14) and avoids frustrating boundary deaths for Tauan.
  • Tiles beyond 20 km may load at coarser LOD (SSE raised to 64 px beyond boundary) to provide visual continuity while minimizing tile cost for out-of-bounds areas.

5. Skybox: CesiumSunSky

The operator's plan of using CesiumSunSky is confirmed as the correct approach. CesiumSunSky in UE5.4 provides:

  • Physically accurate sky given real-world lat/lon and UTC time
  • Automatic sun position calculation (no manual DirectionalLight rotation needed)
  • Seamless integration with Cesium for Unreal tiles (tiles use sun direction for ambient occlusion correctness)
  • Day/night cycle capability (optional for MVP — recommend locking to midday for screenshot-diff determinism per OQ-07)
MVP time-of-day decision (design authority): Fix sun at 12:00 UTC (midday solar time for Inhaúma, MG — approximately 14:00 BRT). This maximizes tile texture visibility (minimal shadow obscuring landmarks), provides the most consistent screenshot-diff baseline, and avoids dynamic sky costs during the performance-constrained MVP phase. Time-of-day progression is a post-MVP feature.

1. Cesium for Unreal v2.x + UE 5.4 Compatibility

CRITICAL COMPAT FINDING: Research on the Cesium for Unreal GitHub releases page (github.com/CesiumGS/cesium-unreal/releases, accessed 2026-05-17) confirms:
  • v2.21.1 (December 2025) is the LAST release to support Unreal Engine 5.4.
  • Release notes for v2.21.1 explicitly state: "This is the last version to support Unreal Engine v5.4."
  • All subsequent releases (v2.22.0+, including current v2.26.0 as of May 2026) require UE 5.5 or later.
  • The discovery report's engine lock (D1: "UE 5.4") therefore pins Cesium for Unreal to exactly v2.21.1 — no newer patch is possible without upgrading UE.
Design implication: The software-architect and devops-engineer must pin CesiumForUnreal at exactly v2.21.1 in the project's UEPlugins.json or equivalent lockfile. Any accidental upgrade will break the UE 5.4 compatibility. This is OQ-10 resolved: version = v2.21.1.

Alternative consideration: Upgrade to UE 5.5 to unlock current Cesium releases (up to v2.26.0 with likely ongoing support). This would require amending D1. The software-architect's report is the appropriate place to opine on this trade-off — game-designer merely notes the dependency constraint.

2. Google Photorealistic 3D Tiles Terms of Service — Game Distribution

Attribution Requirements (Confirmed)

From research on developers.google.com/maps/documentation/tile/policies:

  • The Google Maps logo or the text "Google Maps" must be displayed at minimum 16dp height with 10dp clearance on left, right, and top; 5dp on bottom.
  • All data-provider attributions from tile metadata must be aggregated and displayed "in a line... usually along the bottom of the rendering."
  • For Cesium for Unreal: ShowCreditsOnScreen: true is the exact setting that satisfies this requirement. It must remain enabled in all non-debug builds and in all gameplay screenshots/videos submitted as evidence.
  • Attribution must be visible whenever the map is rendered. It cannot be hidden by HUD elements, post-process overlays, or full-screen effects.

Video Game Use Status

The Google Maps Platform ToS does not explicitly authorize or prohibit use in commercial video games. It permits standard mapping/visualization use. The policy document allows "promotional videos" (max 30 seconds, clearly marked). Game distribution as a downloadable .exe with embedded map tiles constitutes "non-promotional use beyond standard visualization" — which is a gray area requiring legal review or explicit Google sales agreement for the operator's GCP account.

DESIGN RISK — ToS gray area for game distribution: The operator should obtain written confirmation from Google (via GCP support ticket or account manager) that their specific GCP agreement permits embedding Photorealistic 3D Tiles in a downloadable Windows/Linux game binary. Without this, the MVP may be technically in violation of ToS even if the cost is within budget. This risk is listed again in the Design Risks section.

The Cesium World Terrain + Bing Maps fallback avoids this ToS uncertainty entirely (Bing permits integration use; Cesium World Terrain is permissively licensed for developers).

Art Direction Bible

Visual Identity: Photoreal Terrain + Arcade Fighter

The core art direction for aero-fighters-v2 is a deliberate contrast: photoreal world, stylized aircraft. This is not a compromise — it is a design statement. The photorealistic terrain grounds the game in a real place (Inhaúma, MG — a place Tauan knows). The stylized aircraft makes clear "you are the pilot hero, not a real person." This duality matches the arcade-combat identity (D14) while leveraging the full power of the UE5 + Cesium pipeline.

Time of Day

Fixed: Midday (12:00 UTC, approximately 09:00 local Inhaúma time / 14:00 BRT).

Rationale: Midday solar position maximizes tile illumination for landmark recognition, provides the most consistent lighting for screenshot-diff tests (no sun angle variation), and avoids the GPU cost of a full day/night lighting system in the MVP frame budget. CesiumSunSky will be configured with:

SolarTime = 12.0f  (noon UTC)
Latitude   = -19.47f
Longitude  = -44.46f
Date       = fixed arbitrary date (e.g., June 15 — summer/winter consistency irrelevant for MVP)

Weather

Fixed: Clear, no clouds, no precipitation.

Rationale: Clouds add rendering cost (UE5 Volumetric Clouds: ~3–5ms at 1080p); precipitation adds Niagara emitter cost. More critically, clouds obstruct the landmark visibility that screenshot-diff tests depend on. Cloud rendering is a post-MVP visual polish feature.

Inhaúma Landmark Placement

MVP Take-Off / Landing Target: Fazenda Airstrip

There is no public airport within the 20 km box (the two real aerodromes — Fazenda das Perobas at ~17km E and Fazenda do Brejo at ~24km N — are both outside or at the boundary). The MVP airstrip must be a fictional placement consistent with the game-design fiction (see v1 mission-realism SPEC: "the airport is fictional and personal to Tauan").

Design placement decision:
Airstrip coordinates: lat -19.45, lon -44.50 (approximately 4km NW of origin, on flat farmland visible in satellite imagery west of the urban area). This places it in a real field area that appears as agricultural land in photoreal tiles — a believable private farm airstrip. The photoreal tiles will show the actual land here (fields, not tarmac), and the UE5 Blueprint must render the airstrip geometry on top of (overlaying) the tile surface using a flat terrain geometry patch.

The UE5 technique: a flat white-painted StaticMesh (runway) placed at exact terrain height using CesiumGeoreference.InaccurateTransformLongitudeLatitudeHeightToUnreal to convert geo-coordinates to UE world position. This pins the runway to a real geographic point in the photoreal world.

MVP AA Gun Landmark Placement

The AA gun (single static enemy archetype per D8) must be placed at a geographically meaningful position within the 20 km box. The design rationale: the AA gun defends a "strategic" structure. In the Inhaúma real-world context, the most visually recognizable elevated position near the urban core is the church area (a high ground position in Brazilian colonial town planning).

Design placement decision:
AA gun position: lat -19.490, lon -44.387 — approximately 50m east of the church, on the high ground of the town center. This leverages the church spire as a visual reference marker for the player ("the gun is near the church tower").

The AA gun must be placed at exact terrain height using the same geo-coordinate transform technique as the airstrip. It must NOT float above or clip into the photoreal terrain surface — this is the primary placement QA check.

Formal Objection: Upgrade UE to 5.5 (Cross-Domain, for software-architect)

Design objection with evidence (addressed to software-architect + operator):

The operator decision D1 pins UE at 5.4. Evidence now confirms that Cesium for Unreal v2.21.1 is the last version supporting UE 5.4. All subsequent Cesium releases (v2.22.0 through v2.26.0 as of May 2026) require UE 5.5.

Objection: D1 should be reconsidered in favor of UE 5.5, because:
  1. Cesium for Unreal actively develops for UE 5.5+, so bug fixes, tile streaming optimizations, and new terrain features will not reach UE 5.4 builds.
  2. The 4-week timeline (D10) is already high-risk. Debugging Cesium v2.21.1 quirks without upstream support compounds the risk significantly.
  3. UE 5.5 → 5.4 migration risk at project start is lower than staying on a deprecated Cesium integration for the life of the project.
Authority: Per the Decision Authority Matrix, the software-architect has primary authority on architecture decisions; game-developer has override on UE5-specific decisions. This objection is filed for synthesis consideration. If D1 is amended, all downstream version pins must be updated accordingly.

Formal Objection: Screenshot-Diff Baseline License (Cross-Domain, for game-tester)

Design objection with evidence (addressed to game-tester, OQ-06):

The screenshot-diff test harness (AC-V2-06) requires "frozen reference screenshots" for comparison. If these references are captured from the live Google Photorealistic 3D Tiles at a fixed camera pose, they constitute derivative imagery of Google's copyright-protected content.

Objection: Using Google Maps tile imagery as CI test baselines may be a ToS violation ("pre-fetch, index, store, or cache any Content except under limited conditions"). The conditions for CI baselines are not among the explicitly permitted caching exceptions.

Recommendation: Legal review or Google API support ticket before freezing any screenshot-diff baselines. Alternative: use only the engine-rendered game output (which includes the tiles) as the "live expected" and compare against a prior engine build — making the diff a regression test, not a photorealism test.
