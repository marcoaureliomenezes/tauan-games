Architecture Memory — tauan-games

Última atualização: 2026-05-18 · Derivado de: specs/foundation/SPEC.md

Visão geral

O repositório tauan-games é um portfólio de jogos independentes, organizados por uma ladder de quatro engines de complexidade crescente. Cada jogo vive em uma pasta isolada na raiz do repo e compartilha apenas a infraestrutura transversal de testes Playwright (tests/ + package.json) e o diretório de libs vendoradas (vendor/). A arquitetura não tem servidor, build pipeline ou orquestração runtime — os artefatos são index.html + JS estático servidos por npx serve, binários Godot 4 standalone e, no futuro, executáveis UE5 standalone.

Ladder das quatro engines

Toda novidade de jogo é classificada em um destes quatro degraus. Pular degraus ou introduzir uma quinta engine exige decisão explícita do operador via product-engineer.

DegrauEngineVersãoTipoJogo de referência
1Phaser.js3.60 (CDN, sem build)2D webtauan-trex/
2Three.jsr165 (vendor local ES module)3D webaero-fighters/
3Godot 4 (NOVO 2026-05-18)4.x stable point release3D desktop indieaero-fighters-v2/ (UE5 deferred to future release; foundation amended 2026-05-18 per ADR-V2-G-02; hardware-forced from photoreal → stylized → Godot pivot chain)

A ladder é didática: cada degrau ensina conceitos que o anterior não cobre (loop/input/física 2D → modularização 3D + materiais PBR → scene-graph + scripting + signals + export pipeline indie → pipeline industrial AAA).

Emenda 2026-05-18 (ADR-V2-G-02): Ladder ampliada de 3 para 4 degraus. Drive: UE 5.5 source build no Ubuntu 24.04 + Iris Xe é fundamentalmente hostil (clang-18/libicu74 incompatibilidades, DotNet bundled quebrado, GitDependencies droppando binários Linux x86_64 sistematicamente). Godot 4 entra como Degrau 3 (indie 3D desktop) entre Three.js e UE5. UE5 não foi removido; foi deferido para futuro jogo/release. Ver .dadaia/reports/tauan-games/product-engineer/2026-05-18T005834Z-godot-pivot.html.

Estrutura interna por jogo

Degrau 1 (Phaser): index.html + game.js em arquivo único quando possível. README com instruções de execução é obrigatório. Sem build step.
Degrau 2 (Three.js): index.html + src/*.js modularizado em ES modules. Aero Fighters mantém ~15 módulos sob src/ com fronteiras explícitas (main.js apenas orquestra; lógica fica em módulos como player.js, missions.js, airport.js, landing-zones.js, ground-physics.js, sortie-state.js, camera-modes.js, service-scene.js, ejection.js, nuclear-fx.js, debug.js). Vendor local de Three.js em vendor/three.module.min.js.
Degrau 3 (Godot 4): aero-fighters-v2/. Pipeline: scene tree (scenes/*.tscn) + scripts GDScript (com promoção a C# sob triggers de performance) + Resource files (.tres) para config; Autoload singletons para managers globais (MissionManager.gd, GameConfig.gd, Telemetry.gd); Terrain3D addon para heightmap SRTM; export Linux x64 via Godot CLI. Renderer Forward+ (cabe em Iris Xe @ 1080p nativo). Build standalone, distribuído via README link.
Degrau 4 (UE5): reservado para futuro release. Pipeline esperado: Blueprints + C++, build standalone, distribuído via README link. Bloqueado hoje por restrição de hardware (Ubuntu 24.04 + Iris Xe). Será adotado quando hardware permitir (RTX 3060 ou cloud GPU rental).

Fronteiras de autoridade dos agentes

Apenas três agentes podem escrever em repos/tauan-games/. A matriz completa está em .claude/rules/game-agents-coordination.md.

AgenteSub-domínioEscreve
game-developerLógicaC++, JS, Blueprints, IA, física, balística, mecânicas, fixtures de teste
game-designerDesignAssets procedurais, materiais, mapas, áudio, pipeline scripts, specs de assets, HDA
game-testerTestesPlaywright/UE5 automation, reports HTML com evidências

Exceção exclusiva: o product-engineer é guardião absoluto de specs/ em todos os repos e pode editar SPEC/PLAN/TASKS/memory/foundation/constitution sem nunca tocar em código de jogo. Os demais agentes do workspace (software-architect, software-engineer, frontend-engineer, backend-engineer, qa-engineer, devops-engineer) podem ler arquivos para contexto, mas não escrevem.

Fluxo SDD release-based

Releases ativas vivem em specs/releases/<release-id>/ com tripleto SPEC.md + PLAN.md + TASKS.md. O ponteiro do release ativo está em specs/releases/ACTIVE.md:

release: <release-id>
phase: <draft|aprovado|in-progress|closure>

Ao atingir closure, a release migra para specs/_archive/releases/<release-id>/ (read-only). Memory files (este arquivo, tech-stack.html, product/*.html) são atualizados como snapshot atômico do estado pós-closure.

Distribuição

  • Jogos web (degraus 1 e 2): servidos localmente por npx serve; deploy alvo é GitHub Pages (branch gh-pages, cada jogo em subpasta).
  • Jogos Godot 4 (degrau 3): export Linux x64 via godot --export-release; binário standalone único; distribuído via README link. Operator-only local runs em v2 (sem GitHub Releases, itch.io ou Steam).
  • Jogos UE5 (degrau 4, futuro): build pipeline a definir; alvo é executável standalone linkado no README do projeto. Bloqueado por hardware atual.

Invariantes arquiteturais

  1. Nenhum jogo introduz build step, transpilação TypeScript ou bundler nos degraus 1 e 2.
  2. Nenhum jogo carrega assets externos via rede em runtime — todos os sprites, sons e modelos são procedurais ou commitados em vendor/.
  3. Nenhum agente fora do trio game-* escreve em repos/tauan-games/ (exceto product-engineer em specs/).
  4. Toda release passa pela harness Playwright antes de ser considerada CLOSURE.
  5. A ladder das quatro engines é o único conjunto autorizado — quinta engine exige promoção via product-engineer e edição da foundation.
